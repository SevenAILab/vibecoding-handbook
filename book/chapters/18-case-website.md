## 背景：为什么做这个

我做 GEO（Generative Engine Optimization，生成式引擎优化）服务，帮中国出海品牌在 AI 搜索结果里更容易被看到。

这个业务需要一个官网。目标是一个能给客户看的、有信任感的、能收预约的服务介绍站，而不是一个炫酷的技术展示。

目标很简单：客户打开 → 知道我做什么 → 预约一次诊断。

```text
【案例复盘卡：sevenailab 官网】

起点：我需要一个能建立信任、承接预约的 GEO 服务官网
关键动作：用启动 prompt 五要素一次性讲清业务、页面、CTA、语言和审美
最大坑：抽象词太多，AI 会把“自然”“高级”理解成另一套东西
关键判断：官网不是炫技作品，目标是让客户愿意预约一次诊断
可复用动作：先写清楚首屏一句话和 CTA，再让 AI 生成整站
```

## 第一条 prompt

我打开 Lovable，在聊天框里输入了第一条 prompt：

“帮我创建一个为中国出海品牌提供生成式引擎优化（GEO）服务的介绍网站。包含主页、服务介绍、案例研究、工作流程、关于我们、联系我们，和一个‘预约 GEO 诊断报告’的按钮。要求中英双语（英文为主，中文为辅），并参考苹果的极简设计风格。”

这条 prompt 不长，但信息密度很高。拆开看：

```text
【启动 prompt 五要素】

1. 业务定义：给谁、做什么
   → "为中国出海品牌提供 GEO 服务"

2. 页面清单：具体要哪些模块
   → 主页、服务介绍、案例研究、工作流程、关于我们、联系我们

3. 关键动作（CTA）：用户来了之后做什么
   → "预约 GEO 诊断报告"的按钮

4. 语言策略：什么语言、什么比例
   → "中英双语，英文为主"

5. 审美锚点：参考什么风格
   → "参考苹果的极简设计风格"
```

一条 prompt 就把“问题定义”压得很密。Lovable 几十秒后给出了第一版。

## 100 多条消息的迭代

第一版出来后，接下来两天就是不断迭代。

我没有一次性列出所有修改，而是一步步来。看到什么问题，描述给 AI，让它改。看改完的效果，再找下一个问题。

举几条顺利的修改指令：

“首页的标题太大了，改小一点。”
“服务介绍部分用三列卡片排列，每张卡片有图标、标题和一句话说明。”
“CTA 按钮改成蓝底白字，放在首屏可见区域。”

这类指令一条就过，因为意思明确，AI 没有歧义空间。

但 100 多条消息里，至少有三分之一不是一次成功的。

**“关于我们”的文案改了四次。** 第一次我说“改得自然一点”，AI 把公司介绍变成了一段鸡汤。第二次我说“像一个人在说话，不像公司介绍”，它变成了一段自嗨式的创业故事。第三次我换了策略，直接给了一段我自己写的原文，让 AI 润色。结果它“润”得面目全非，把我的口语全改成了书面语。第四次我学聪明了，给了原文，加了一条限制：“只改语法错误，不要改语气和用词。”这才对。

回头看，问题出在“自然”这个词太模糊了。AI 对“自然”的理解和你对“自然”的理解完全不同。你得给参照物，或者直接给原文让它微调，而不是让它从零重写。

**案例研究的数据图做了三版。** 第一次我说“加一个前后对比的数据图”，AI 做了一个柱状图。技术上没错，但放在官网上像一份内部报告。第二次我说“不要图表，用数字对比的方式呈现”，它做了一个纯文字列表，没有视觉冲击力。第三次我截了一个竞品网站上类似的“before/after”对比区块，发给 AI 说“参考这个排版”，一次就对了。

这印证了 第六章 说的：当你用文字描述不清楚的时候，截图比打字管用一百倍。

**手机端排版翻车了两次。** 我一直在电脑上看效果，直到快完成时用手机打开，才发现服务介绍的三列卡片挤成了一坨，CTA 按钮被挡在屏幕外面。第一次我说“手机上排版要正常”，AI 改了卡片布局，但按钮还是在很下面的位置。第二次我具体说“手机端卡片改成单列，CTA 按钮在首屏就要能看到”，才解决。如果我从第一天就用手机检查，这两轮来回可以省掉。

100 多条消息里，真正“一条就过”的大概只有一半。剩下的要么是描述不够具体导致 AI 理解错了，要么是我自己也没想清楚想要什么，改着改着才找到方向。这个过程不丢人，做产品本来就是这个样子。

## 一键数据库

做到联系表单的时候，我需要一个地方存用户提交的预约信息。在 Lovable 的“云（Cloud）”面板，点“启用”。它自动创建了数据库和自动化流程：有新表单提交时，自动发邮件到我的邮箱。这就是 Lovable 的一键化范式。

---

## 完整复盘

两天、100 多条消息。整个过程就是：第一条启动 prompt 定义清楚 → 看到成品 → 每次发现一个问题，用自然语言描述给 AI → AI 修改 → 看效果 → 再找下一个问题。

没有任何技术配置。没有代码。流程足够短到可以在两天内完成。这个案例教我的最重要的一点是：启动 prompt 的质量决定了一切后续的工作量。

---

## 这个案例教了我什么

回头看 sevenailab 这个项目，它验证了几件事：

第一，启动 prompt 的质量决定了第一版的质量。五个要素越清楚，AI 第一版给你的越对，后面迭代的次数越少。

第二，Lovable 的第一版是一个 80 分的完整产品，不是空白模板。你直接在上面精修，省掉了搭脚手架的阶段。

第三，非技术人员最怕的几件事，比如数据库、域名、API，现在都有托管平台或低代码方案可以帮你简化。你不需要精通，但要知道这件事大概在哪一层发生。

第四，两天做完一个官网的前提是：你在第一条 prompt 之前就想清楚了“这是给谁的、做什么的、用户来了之后做什么”。磨刀不误砍柴工。

---

## 要点小结

官网从零到上线，难点不在技术能力，而在第一条 prompt 的业务定义。启动 prompt 的五个要素越清晰，AI 生成的第一版越接近目标，迭代轮次越少。sevenailab 用两天和 100 多条修改指令完成了全流程。最后一公里的域名、数据库、支付这些“烦人的事”，现在也有很多托管方案能简化，但仍然需要你理解基本链路并做验收。

**本章行动**：用启动 prompt 五要素写一条属于你自己项目的启动 prompt，发给 Lovable。观察第一版出来的样子，记录你需要修改的地方。然后花一个小时修改 5-10 处细节。这不是完整官网的要求，是“能给真人看”的标准。
