---

做 sevenailab 的时候，我想加一个联系表单。用户填姓名、邮箱、需求，提交后我能看到。

听起来简单。但这里有一个问题：用户提交的数据需要存在某个地方。

它不在用户的浏览器里（刷新就没了），也不在我的电脑里（用户又不在我电脑上），而在一个“云端仓库”里。这个云端仓库就是数据库。

在 Lovable 里，我启用了数据库功能（不同版本里的入口名称可能会变化，当时是在“云（Cloud）”面板）。它自动创建了数据库，用来储存表单提交的数据，还能设一个自动化流程：当有新表单提交时，自动发一封邮件到我的邮箱。

过去要走后端 + API + 数据库三步，现在一次点击就能完成。

但如果你用 Cursor，或者你的需求比一个表单更复杂（比如用户登录、保存用户数据、管理订单），你需要理解数据库到底是什么，以及怎么接一个。

这一章信息量会比前面大。建议你分两次读：第一次读到“接入 Supabase 的最小路径”结束，只理解数据库和存数据；第二次从“用户登录”开始，理解登录、权限和环境变量。不要指望一口气全懂，能把关键概念抓住就够了。

## 数据库是什么

数据库，用最朴素的话说，就是一个有结构的大表格。

想象一个 Excel。每一张工作表是一个“表”（Table），每一列是一个“字段”（Column），每一行是一条“记录”（Row）。

比如你有一个“用户”表：

```text
用户表（users）
┌─────┬──────────┬────────────────────┬────────────┐
│ id  │ name     │ email              │ created_at │
├─────┼──────────┼────────────────────┼────────────┤
│ 1   │ 张三     │ zhang@example.com  │ 2026-04-01 │
│ 2   │ 李四     │ li@example.com     │ 2026-04-02 │
│ 3   │ 王五     │ wang@example.com   │ 2026-04-03 │
└─────┴──────────┴────────────────────┴────────────┘
```

就这么简单。

数据库和 Excel 的区别在于：它能被你的网站代码读写，能处理成千上万条数据而不卡，能设置权限（谁能看、谁能改），能和其他表建立关系。




```text
【数据库四概念】

1. 表（Table）
   → 一类数据的集合，像 Excel 的一张工作表
   → 例：users 表、orders 表、posts 表

2. 字段（Column / Field）
   → 数据的类型，像 Excel 的列标题
   → 例：name、email、price
   → 每个字段有类型：文字（text）、数字（integer）、日期（timestamp）

3. 记录（Row / Record）
   → 一条具体的数据，像 Excel 的一行
   → 例：张三的信息就是一条记录

4. 主键（Primary Key）
   → 每条记录的唯一身份证号
   → 通常是 id 字段，自动递增，不会重复
```

你不需要会写 SQL（数据库的操作语言）。但你要知道这四个概念。因为当你告诉 AI“帮我建一个用户表”时，它会问你“需要哪些字段？”你得能回答。

SQL 的常见操作也就四个：增（INSERT，插入新数据）、删（DELETE，删除数据）、改（UPDATE，修改数据）、查（SELECT，查找数据）。AI 会帮你写 SQL，但你要知道你在让它做哪一种操作。

## Supabase：你的数据库首选

数据库有很多选择。但如果你是个人开发者、做第一个有数据的产品，我推荐 Supabase。

Supabase 是开源的 Firebase 替代品。它给每个项目提供一个完整的 PostgreSQL 数据库，加上用户认证、文件存储、实时推送等功能。

为什么推荐它？

第一，起步门槛低。几分钟内就能创建一个可用的数据库项目。注册 → 新建项目 → 等它初始化，就有了。

第二，早期成本低。你的产品在早期阶段，Supabase 的免费或低价额度通常够用，具体额度以上架时官方套餐为准。

第三，功能比较全。数据库、身份验证、存储、实时、邮件功能都有。你不需要从五个不同的服务商分别接入。

第四，不是黑盒。你可以在 Supabase 后台直接看到你的数据表、直接写 SQL 查数据、直接导出。数据在你手里。

初期你只需要用它的两块能力：Database（存数据）和 Authentication（用户登录）。其他能力以后用到了再学。

## 接入 Supabase 的最小路径

怎么在你的项目里接入 Supabase？

先记住一句话：数据库解决“存数据”，登录解决“这是谁的数据”。

如果只是一个公共留言表单，所有人提交的信息都进同一张表，你暂时可以不登录。但只要涉及“我的历史记录”“我的订单”“我的草稿”“我的积分”，你就需要登录。否则系统不知道这条数据属于谁。

第一版只要做到四件事：




```text
【数据库 + 登录最小目标】

1. 能建一张表
2. 能把一条用户输入存进去
3. 登录后只看到自己的数据
4. Key 不泄露
```

其他高级功能，比如权限系统、头像上传、团队成员、复杂表关系，都可以后面再说。

如果你用 Lovable，最简单：在云面板里点启用。不同版本的入口名称可能变化，但大体思路是让 Lovable 帮你创建或连接后端数据库服务。

如果你用 Cursor，需要多几步，但跟着走就行：

```text
【接入 Supabase 的具体步骤】

第 1 步：打开 supabase.com，注册账号（用 GitHub 登录最快）。

第 2 步：点"New Project"，填项目名称，选一个离你用户近的服务器区域，
       设置数据库密码（记下来），点"Create new project"。
       等 1-2 分钟让它初始化。

第 3 步：进入项目后，点左边菜单的"Table Editor"。
       点"New Table"，给表起个名（比如 contacts）。
       添加字段：name（类型选 text）、email（类型选 text）、
       message（类型选 text）。点"Save"。

第 4 步：点左边菜单的"Settings" → "API"。
       你会看到 Project URL，以及用于前端访问的 publishable key 或 public/anon key。
       Supabase 后台的 key 命名可能会调整，以当前页面说明为准。
       复制这两个值。

第 5 步：在你的项目文件夹里创建一个叫 .env.local 的文件，写入：
       NEXT_PUBLIC_SUPABASE_URL=你的 Project URL
       NEXT_PUBLIC_SUPABASE_ANON_KEY=你的 public/anon key 或 publishable key

第 6 步：在 Cursor 里告诉 AI：
       “帮我接入 Supabase，URL 和 public/anon key（或 publishable key）在 .env.local 文件里。
        我要把联系表单的数据存到 contacts 表。
        请确认公共表单只能提交，不能读取别人的提交。”
       AI 会帮你安装 Supabase 的库并写好连接代码。
```

还有一个可选路线：如果你用 Vercel 部署，也可以在 Vercel 的 Storage / Marketplace 里选择当前支持的数据库服务（例如 Neon Postgres 等）。Vercel 会把连接信息和环境变量配置流程做得更顺。具体入口和合作服务会变化，按 Vercel 后台当前提示操作即可。它适合只需要简单存数据、且想把部署和数据库放在同一套平台里管理的场景。

AI 会帮你安装 Supabase SDK、配置连接、写数据读写的代码。

一个重要的提醒：调用 API 的标准流程是，阅读 API 文档 → 获取 URL 和 key → 让 AI 学习文档 → 描述需求 → 配置存 `.env.local` 或部署平台环境变量 → 调试。secret key、service role key、支付密钥这类高权限配置永远不要写在公开代码中，也不要上传到 GitHub。能放在前端的公开 key 也不等于“没有风险”，仍然要确认用户只能访问自己该访问的数据。

在 Vercel 部署时，环境变量需要在 Vercel 后台单独配置（Settings → Environment Variables）。这些变量在构建时注入，比在代码中写死安全得多。

有人把 AI 模型的 secret key 直接写在代码里推上了 GitHub，3 小时内被爬虫扫到，一天之内被刷了上千次调用，账单翻了 10 倍。secret key 泄露是真金白银的教训。

## 用户登录

数据库解决“存数据”的问题。登录解决“谁的数据”的问题。




没有登录，所有用户看到的是同一份数据。有了登录，每个用户看到的是自己的数据：自己的收藏、自己的设置、自己的订单。

登录方案有三层递进：




```text
【登录方案三层递进】

第一层：Supabase Auth（推荐新手）
  → 全功能、文档完整
  → 支持 GitHub/Google 社交登录
  → 很多成熟独立产品也会继续使用
  → 和 Supabase 数据库天然集成

第二层：Clerk
  → 登录体验更好、UI 开箱即用
  → 但只有登录功能，没有数据库
  → 适合对登录体验要求高的产品

第三层：NextAuth / BetterAuth
  → 功能完整但配置复杂
  → 适合有特殊需求的高级场景
```

建议从 Supabase Auth 开始。它和你的数据库在同一个平台，集成最简单。

一个实操建议：如果你的用户主要在海外，优先用 GitHub 或 Google 社交登录。用户点一下就登进来了，你不需要做忘记密码、修改密码这些东西。

但如果你的产品面向中国用户，Google 社交登录基本不可用。大部分用户没有 Google 账号，或者网络环境打不开 Google 的授权页面。

这时候用 **Magic Link 邮箱登录**。

Magic Link 的流程是这样的：用户输入邮箱 → 点“发送验证码” → 去邮箱里收一封验证邮件，里面有一个验证链接或验证码 → 点击链接或输入验证码 → 登录成功。不需要设密码，不需要记密码。每次登录都是“发一封邮件验证一下”。

Supabase Auth 原生支持 Magic Link。在 Cursor 里告诉 AI：“用 Supabase Auth 的 Magic Link 方式做邮箱登录，用户输入邮箱后发送验证码，验证通过后登录。”AI 会帮你写好整个流程。

在 Supabase 后台配置也不复杂：进入 Authentication → Providers → Email，确认 Email provider 是开启的，Magic Link 选项是打开的。Supabase 会自动帮你发验证邮件，你不需要自己搭邮件服务器。

简单记住这条就行：海外用户用 Google/GitHub 社交登录，国内用户用 Magic Link 邮箱登录。两种方案 Supabase 都支持，选哪种取决于你的用户在哪里。

如果你用 Next.js 做带登录的产品，Supabase 官方推荐使用 `@supabase/ssr` 这类服务器端认证方式，而不是把所有认证逻辑都放在客户端。这个细节对安全性很重要。你不需要一开始理解所有原理，只需要在让 AI 帮你接入登录时提一句：“请按 Supabase 当前官方文档，用 `@supabase/ssr` 做服务器端认证。”

还有一个安全检查必须记住：用户只能看到自己的数据。比如用户 A 登录后，只能看到自己的历史记录、订单和草稿，不能看到用户 B 的。公共表单可以允许陌生人提交，但不应该允许陌生人读取别人的提交。你不用自己写复杂权限规则，只要在让 AI 接入登录或数据库时补一句：“请确认登录用户只能访问自己的数据；如果是公共表单，只允许提交，不允许读取其他人的提交。”

## 登录后的用户管理

用户通过 GitHub 或 Google 登录成功后，Supabase Auth 会创建一条认证用户记录。你的业务数据库里是否还需要一张 `profiles` 或 `users` 表，要看产品需求，有时需要额外设计或用触发器同步。登录成功后，用户可以被重定向到 dashboard 页面。你在 Supabase 后台可以看到认证用户信息。

后续你可以根据用户 ID 做数据关联，比如“这个用户收藏了哪些内容”“这个用户的订单记录”。

但这些都是后面的事。第一步只要做到：用户能登录、登录后能看到自己的页面。这就够了。

## 环境变量：你一定会在这里卡住

环境变量是这一章（甚至整本书）最容易让人崩溃的环节。我自己在上面花了非常多时间。所以这里写得详细一点。

环境变量是什么？就是一组“配置”：你的 Supabase URL、公开访问 key、AI 模型 secret key、Stripe 密钥等等。其中 secret key 和支付密钥不能写在公开代码里；公开访问 key 可以给前端用，但仍然要配合权限检查。为了统一管理，第一版都先放在单独配置里。

你可以把环境变量想成手机通讯录。App 需要打电话时，不会把号码写死在每一段代码里，而是去通讯录里查。`.env.local` 就是你项目的通讯录：代码只知道“去找 SUPABASE_URL”，真正的地址和 key 放在这个文件里。

你需要在两个地方配置环境变量：本地开发时的 `.env.local` 文件，和部署平台（Vercel）的后台。两边要一模一样。




```text
【环境变量配置：完整步骤】

第 1 步：本地创建 .env.local 文件

  在你的项目根目录（就是最外层的文件夹）里，
  创建一个叫 .env.local 的文件。
  注意：文件名以点号开头，没有其他扩展名。

  在 Cursor 里直接告诉 AI：“帮我创建 .env.local 文件。”
  或者自己新建文件，文件名填 .env.local。

第 2 步：填入 Supabase 的值

  打开 Supabase 后台 → Settings → API。
  你会看到两个值：
    - Project URL（长得像 https://xxxxx.supabase.co）
    - publishable key 或 anon public key（一长串字母数字，以后台当前命名为准）

  把它们写进 .env.local：

  NEXT_PUBLIC_SUPABASE_URL=https://xxxxx.supabase.co
  NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOiJIUzI1NiIsInR5cCI6...

  等号两边不要有空格。值不需要加引号。变量名如果用 `NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY` 也可以，但要和代码里读取的名字一致。

第 3 步：确认 .gitignore 包含 .env.local

  这一步是防止你的钥匙被上传到 GitHub。
  打开项目里的 .gitignore 文件，确认里面有这一行：

  .env.local

  如果没有，加上去。或者告诉 AI：
  “确认 .gitignore 里包含了 .env.local。”

第 4 步：在 Vercel 后台配置同样的变量

  这是大部分人卡住的地方。
  本地 .env.local 只在你自己电脑上生效。
  部署到 Vercel 后，Vercel 的服务器看不到你电脑上的文件。
  所以你必须在 Vercel 后台再填一遍。

  打开 vercel.com → 进入你的项目 → Settings → 
  Environment Variables。

  逐条添加：
    Key 填：NEXT_PUBLIC_SUPABASE_URL
    Value 填：和 .env.local 里一样的值
    环境选：Production + Preview + Development 全勾上

  再添加：
    Key 填：NEXT_PUBLIC_SUPABASE_ANON_KEY
    Value 填：和 .env.local 里一样的值

  如果你的项目使用的是 NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY，
  就按代码里的变量名填写，关键是名字和值要对应。

  保存后，点 Deployments → 找到最新的部署 → 
  点右边三个点 → Redeploy。

  重新部署后新的环境变量才会生效。

第 5 步：验证

  部署完成后，打开你的网站，试一下需要数据库的功能
  （比如提交表单、登录）。
  如果能正常工作，环境变量就配对了。
  如果报错，优先检查这几类问题：
    - Vercel 里漏了某个变量
    - 变量名拼写和代码里不一致（多了空格、少了字母）
    - 添加完变量后没有重新部署
```

后面当你接入 Stripe 支付、AI API 等服务时，流程类似：从服务商后台拿到 key 或密钥，写进 `.env.local`，再填进 Vercel 后台。需要给浏览器用的变量通常会带 `NEXT_PUBLIC_` 前缀；真正的 secret key 不要加这个前缀，只能在服务端使用。这个操作你会越来越熟练。

记住一个原则：如果你的产品“本地能跑，线上不行”，第一反应就是检查环境变量。它是最常见的线上问题之一。

## 要点小结

数据库让你的产品从“无状态”升级到“有状态”。个人开发者做第一版时，Supabase 是很顺手的起点：免费额度够早期使用，数据库、登录、存储都在一个平台里，数据也能在后台直接看到。登录方案看用户在哪里：海外用 Google/GitHub 社交登录，国内用 Magic Link 邮箱登录。环境变量是最容易卡住的环节，本地 .env.local 和 Vercel 后台两边都要配，配完要重新部署。

**本章行动**：在 supabase.com 创建项目，在 Table Editor 创建一个简单的表（name、email、message 三个字段）。如果用 Lovable 就在云面板启用；如果用 Cursor，就获取 URL 和公开访问 key 存进 `.env.local`，让 AI 按官方文档帮你接入，并检查“用户只能访问自己的数据，公共表单只能提交不能读取别人提交”。
