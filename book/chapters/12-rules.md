# 第十二章 · 协议层：prompt 是临时话，Rules 是长期合同

> 所属部分：第三部分 · 进阶，把工具用深  
> 读完本章，你会得到：一份项目 Rules 文件  
> 本章方法：Rules 三层结构 + prompt 升级四问

---

你有没有发现一个问题？

每次打开 Lovable 或 Cursor 开始新的对话，你都在重复说同样的话。

“我的项目是一个 GEO 服务官网，面向中国出海品牌。”“用 Tailwind CSS 和 shadcn/ui。”“风格要克制、不要紫色。”“按钮用蓝色，圆角 8px。”

你说了十遍了。AI 每次都忘。

这不是 AI 记性差。是因为你一直在用“临时话”和它沟通。每次聊天开始，上下文清零，它什么都不记得。

这一章解决这个问题。我们把“临时话”变成“长期合同”：一份写在文件里的规则，AI 每次启动都会自动读取。

---

## 两种和 AI 沟通的方式

你和 AI 之间其实有两种沟通方式。

第一种是 prompt，也就是你在聊天框里打的每一句话。它是临时的、一次性的。新的对话开始，之前说的就没了。

第二种是 Rules，写在项目文件里的规则。AI 每次打开项目时自动加载，不需要你重复说。

prompt 像口头约定。“这次帮我用蓝色做按钮。”下次你不说，它可能就用回默认的紫色了。




Rules 像书面合同。“所有按钮用蓝色、圆角 8px、hover 时加深。”写一次，每次生效。

你在做第一版的时候，prompt 就够用了。需求简单、改动不多、几句话能说清。

但当你的项目开始变复杂，有十几个页面、有固定的设计风格、有一套你反复使用的技术栈，你会发现自己在重复劳动。每次新对话都要重新建立上下文。

这时候就该写 Rules 了。

---

## 不同工具的 Rules 长什么样

三个工具，三种项目规则写法。名字不同，作用接近。




**Lovable**：没有独立的 Rules 文件，但它有 prompting Best Practices。官方建议在写 prompt 前做四个问题的规划：这是什么产品？谁会用？为什么用？用户的关键行动是什么？你把这四个答案写在每次对话的开头，就是最简单的 Rules。

**Cursor**：优先使用 `.cursor/rules` 目录来管理项目规则；旧项目里也可能看到 `.cursorrules` 文件，那是早期写法。它们存在你的项目文件夹里，Git 可追踪，团队可共享。Cursor Agent 在合适的会话中会参考这些规则。具体文件格式和生效方式以 Cursor 当前文档为准。

**Claude Code**：用 `CLAUDE.md` 文件。放在项目根目录，作为项目说明和工作约定。可以用它记录编码标准、架构决策、库偏好、review 清单、build 命令。

不管哪种工具，Rules 做的事情是一样的：让 AI 在开始工作前就知道“这个项目的规矩是什么”。

---

## Rules 三层结构

一份好的 Rules 文件应该包含什么？我把它分成三层：

```text
【Rules 三层结构】

第一层：项目身份（你是谁）
  - 产品是什么、给谁用、核心价值
  - 用了哪些技术栈（Next.js / Tailwind / Supabase...）
  - 参考审美和设计原则

第二层：代码规范（怎么写）
  - 代码风格（函数式、模块化）
  - 命名规范（小写+连字符 / 驼峰命名）
  - TypeScript 用法（interface 优先 / 严格类型）
  - UI 规范（移动优先、用 shadcn/ui 组件）

第三层：行为约束（不做什么）
  - 不要在没问我的情况下删除已有功能
  - 不要用行内样式，用 Tailwind 类名
  - 修改前先说明你要改什么，等我确认
  - 不要虚构 API 端点
```

第一层确保 AI 理解上下文。第二层确保代码一致性。第三层避免常见的 AI 失控行为。




来自 cursor.directory 的一份推荐配置是这样的：针对 Next.js + TypeScript + shadcn/ui 项目，Rules 应包括代码风格（函数式、模块化）、命名规范（小写+dash）、TypeScript 使用（interfaces 优先）、UI 设计（shadcn + Tailwind、移动优先）、性能优化（最小化 use client）。

你不需要从零写这些。你可以让 AI 帮你写。

## 让 AI 帮你写 Rules

我做伴聊的时候，第一件事就是让 Claude 帮我生成 Cursor Rules。我的 prompt 是这样的：

“我要用 Cursor 开发微信小程序，请帮我生成一份 Cursor Rules。要求包括：代码风格规范（命名、注释、缩进）、小程序开发的最佳实践、错误处理原则、代码审查检查项、针对微信小程序的特殊要求（如云函数调用、页面生命周期等）。请优先按 `.cursor/rules` 的方式组织，如果当前项目更适合旧式 `.cursorrules`，也请说明。”

Claude 给我生成了一份几十行的规则文件。我复制粘贴到项目根目录，从此 Cursor 每次打开项目都会按这些规则工作。

这个文件确保 AI 写出的代码高度规范。更重要的是，后续维护也方便：任何人（或 AI）接手这个项目，读一遍 Rules 就知道该怎么写代码。

你也可以这样做。把下面这个 prompt 模板填上你的项目信息，发给 AI：

```text
我要用 [Cursor/Claude Code] 开发 [你的产品类型]，
技术栈是 [Next.js + TypeScript + Tailwind + shadcn/ui + Supabase]。
请帮我生成一份 [.cursor/rules / CLAUDE.md] 文件。
要求包括：
1. 项目背景和核心功能描述
2. 代码风格规范
3. UI/UX 设计原则
4. 错误处理规则
5. 不应该做的事情（行为约束）
请直接输出可以复制粘贴使用的文件内容。
```

## Cursor Rules 的四种绑定方式

Cursor 的 Rules 不只是“一个文件”。它支持四种应用策略，你可以按需组合：

```text
1. Always Apply（总是应用）
   → 全局约束，每次 Session 都生效
   → 适合：代码风格、命名规范、技术栈约束

2. Apply Intelligently（智能应用）
   → Agent 根据规则的描述自动判断是否相关
   → 适合：特定模块的开发规范

3. Apply to Specific Files（文件级应用）
   → 用 glob 模式匹配特定文件
   → 适合：对某一类文件（如 *.test.ts）施加特殊规则

4. Apply Manually（手动应用）
   → 在聊天中用 @mention 手动触发
   → 适合：偶尔需要的特殊规范
```

对你来说，大部分情况用第一种“Always Apply”就够了。把你的项目规范写在一个 Rules 文件里，它每次都会读。

## CLAUDE.md：Claude Code 的记忆

如果你开始用 Claude Code，CLAUDE.md 就是你的 Rules 文件。

它的作用和 Cursor Rules 类似，但 Claude Code 的设计哲学更强调“对话感”。你可以用自然语言描述你的项目和规则，而不是写复杂配置。

比起写“写单元测试”这种技术命令，Claude Code 更适合你说“为 auth 模块写测试，跑一遍，修复失败”。具体业务问题（“用户登录后看到空白屏”）比泛泛的“修 Bug”更容易定位。

CLAUDE.md 里可以写什么？编码标准、架构决策、库偏好、review 清单、常用 build 命令。把这些东西集中写在一个文件里，避免了在每次 chat 中重复说明。AI 每次启动都会读这个文件，直接进入状态。

如果你不知道 CLAUDE.md 该怎么写，可以参考公开模板或让 AI 根据你的项目生成初版。不要直接照搬别人的规则文件；更好的做法是把模板当成清单，留下适合你项目的部分，删掉你不理解或用不上的部分。

## prompt 也可以升级

不是所有场景都需要 Rules 文件。你在 Lovable 里、或者做一次性修改时，prompt 仍然是主要沟通方式。

但 prompt 也有好坏之分。

我总结了一个“prompt 升级四问”。在你发送任何 prompt 之前，用这四个问题检查一下：




```text
【prompt 升级四问】

1. 有没有上下文？
   → “改一下按钮” vs “首页的 CTA 按钮，在移动端看不到，需要调整位置和大小”
   
2. 有没有参照物？
   → “做好看一点” vs “参考 Stripe 的定价页风格”

3. 有没有约束？
   → “帮我改” vs “只改 CSS，不要动 JavaScript 逻辑”

4. 有没有验收标准？
   → “优化一下” vs “优化后移动端加载时间低于 3 秒”
```

上下文让 AI 知道你在说哪个部分。参照物让它知道你想靠近什么。约束让它知道不能碰什么。验收标准让你知道怎么判断“做完了”。

还记得 第二章的描述三件套吗？prompt 升级四问是同一套思路，只是对象从“整个产品”变成了“某一个功能”。

## 系统逻辑梳理：一个被低估的用法

Rules 和 prompt 都是“你告诉 AI 做什么”。

但有一种用法是反过来的：让 AI 告诉你“系统是怎么连接的”。

我做 7brandhub 的时候用过一个很有效的 prompt：

“请帮我详细解析开发这个网站的完整逻辑链条。从在 Namecheap 买域名，到用代码工具写代码，再到推送到 GitHub、导入 Vercel、链接数据库，它们之间到底是怎么串联的？”

AI 把整个部署链路拆成了一张流程图。我瞬间看清了哪一步和哪一步是连着的，出了问题该查哪里。

这个用法你可以在任何阶段使用。当你觉得“项目越来越复杂，不知道什么和什么在连接”的时候，让 AI 帮你画一张系统逻辑图。这比死磕代码有效得多。

## 从口头约定到书面合同

这一章其实就一条线：

你和 AI 的沟通方式，要从“每次口头重复”进化到“一次写好，长期生效”。

Lovable 阶段，你用 prompt 就够了。但要用好，每次发送前过一遍“四问”。

Cursor 阶段，写一份项目 Rules。让 AI 帮你生成初版，你调整后放进 `.cursor/rules` 或工具当前推荐的位置。

Claude Code 阶段，写一份 CLAUDE.md。把项目背景、编码规范、行为约束全部写进去。

这些文件做的事情，本质上就是一件：减少沟通成本。

你写需求的时间是固定的。但重复说明的时间可以被一个文件消除。

再往前走一步，Claude Code 还有一个叫 Skills 的概念：如果说 Rules 是“这个项目的合同”，Skills 就是“可以带到下一个项目的可复用技能包”。这个留到附录 D 再展开，现在知道有这个东西就行。

---

## 要点小结

从“每次重复说一遍”进化到“一次写好，长期生效”。prompt 是临时话，Rules 是长期合同。当项目开始稳定下来，就应该把风格、技术栈、行为规范写成 Rules 文件，减少沟通成本。

**本章行动**：用本章的 prompt 模板，让 AI 帮你生成一份适合你项目的 Rules 文件（如 `.cursor/rules` 或 CLAUDE.md），放进项目目录。下次发修改指令前，用“prompt 升级四问”快速检查一遍。
