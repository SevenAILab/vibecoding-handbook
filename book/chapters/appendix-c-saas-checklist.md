## 一、Supabase 能力选择表

```text
第一版通常需要：
□ Auth：注册、登录、找回账号
□ Database：用户数据、产品数据、订单数据
□ 权限检查：用户只能访问自己的数据；公共表单只能提交，不能读取别人提交

按需再加：
□ Storage：头像、图片、附件上传
□ Edge Functions：支付回调、后台调用 AI、发邮件
□ Realtime：聊天、协作、实时通知

给 AI 的提示：
“请先只接入 Auth + Database，并确认用户只能访问自己的数据。
其他能力先不要加，后面需要时再扩展。”
```

## 二、支付链路的四个角色

```text
角色 1：定价者 — 谁来决定价格？
  → 你在支付平台后台设置价格（Stripe Dashboard / 微信支付后台）
  → 或者你在自己的数据库里存价格，前端动态展示
  → 第一版建议：直接在支付平台后台设定，最简单

角色 2：确认者 — 谁来确认钱到了？
  → 用户点了"支付"，支付平台处理扣款
  → 扣款成功后，支付平台通过 Webhook 通知你的服务器
  → Webhook 就是"支付平台主动给你发一个消息"
  → 你的服务器收到消息、验证签名、确认订单状态后，才算"确认支付成功"

角色 3：更新者 — 谁来更新用户权限？
  → 确认支付成功后，你的后端代码更新数据库
  → 比如把 users 表里该用户的 plan 字段从 "free" 改成 "paid"
  → 或者把 subscription_end 日期设成一个月后
  → 前端根据这个字段判断用户能用哪些功能

角色 4：回收者 — 谁来回收过期权限？
  → 订阅到期后，用户的付费权限需要被收回
  → Stripe 会在订阅到期时自动发 Webhook 通知你
  → 你的后端代码把 plan 改回 "free"
  → 微信支付没有自动订阅管理，需要自己做定时检查
```

底线：支付成功不能只看前端跳转。真正的权限开通，应该由支付平台回调确认后再更新数据库。

## 三、收钱路径对照表

下面这张表把国内和海外两条路线的关键环节放在一起对比，方便你根据自己的情况选择：

```text
环节              国内路线                    海外路线
主体资质          营业执照（个体户/公司）      公司注册（香港/美国/新加坡）
支付方式          微信支付 / 支付宝            Stripe
用户付费习惯      扫码支付、小额高频           信用卡、订阅制
审核周期          通常数个工作日，视类目而定     视地区、主体和材料而定
订阅管理          需要自己设计或接服务商能力     Stripe 提供订阅能力，但仍要处理 Webhook
测试方式          沙箱环境或小金额实测         测试卡号 4242 4242 4242 4242
资金到账          以商户后台规则为准            以 Stripe 当前结算规则为准
最低门槛方案      微信转账（手动验证）          第三方售卖平台或可注册 Stripe 的主体
```

## 四、没有主体资质时的替代路径

```text
□ 微信转账
  适合：服务、诊断、人工交付
  优点：最快验证有没有人愿意付费
  缺点：完全手动，不适合规模化

□ 微信小商店
  适合：模板、资料包、轻服务
  优点：门槛较低
  缺点：仍要看平台规则

□ 知识星球 / 小报童
  适合：内容、社群、服务包
  优点：不需要自己开发支付系统
  缺点：更像卖内容或服务，不像完整 SaaS

□ Gumroad / Lemon Squeezy 等海外售卖平台
  适合：英文市场、模板、数字产品
  注意：可用地区、税务、身份要求和抽成规则会变化，以上架时平台政策为准。
```

## 五、成本底线：别卖一单亏一单

如果你的产品调用 AI 模型，每次使用都有成本。在定价之前，先估算一下：

```text
单次使用成本 = 输入 token 数 × 单价 + 输出 token 数 × 单价

举例（按当时常见模型价格粗估，具体以服务商当前价格为准）：
  一次 AI 对话约 2000 输入 token + 1000 输出 token
  成本约 ¥0.05-0.10
  如果用户每天用 10 次 = 每天 ¥0.5-1.0
  一个月 = ¥15-30

如果你的月订阅定价是 ¥19.9，成本可能占到 75%-150%。
这意味着：用户用得越多，你亏得越多。
```

解决办法：限制免费用户的使用次数（比如每天 3 次），或者选择更便宜的模型（比如 GPT-4o-mini），或者提高付费档的价格。

不需要做成精确的财务模型。但至少要心里有数：你的定价能不能覆盖 AI 调用成本。如果覆盖不了，要么调价格，要么调模型，要么调使用量上限。

## 六、从“能用”到“能卖”的检查清单

```text
功能层：
  □ 核心功能跑通
  □ 四种状态都有（Loading / Error / Empty / Success）
  □ 移动端能用

用户层：
  □ 用户能注册和登录（Supabase Auth）
  □ 权限检查过（用户只能看到自己的数据；公共表单不能读取别人提交）
  □ 数据能保存（用户回来能看到历史）

商业层：
  □ 定价方案确定（免费试用 + 付费升级）
  □ 支付链路四个角色都想清楚了
  □ 有付费入口（哪怕只是一个"升级"按钮）
  □ 成本底线算过（不会卖一单亏一单）

合规层：
  □ 服务条款页面
  □ 隐私政策页面
  □ 退款政策页面

运营层：
  □ 有反馈渠道（用户能告诉你哪里不好用）
  □ 能部署到公网
  □ 统计代码接好了（知道有没有人来）
```

第一版不需要全部打勾。但你要知道完整的闭环长什么样。缺的每一项都是从“能用”到“能卖”之间的一个坑。提前知道有这些坑，总比踩进去才发现要好。
