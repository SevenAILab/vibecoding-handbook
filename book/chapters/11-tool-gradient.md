# 第十一章 · 工具梯度：Lovable → Cursor / Codex → Claude Code

> 所属部分：第三部分 · 进阶，把工具用深  
> 读完本章，你会得到：个人工具选型决策表  
> 本章方法：工具梯度 + 升级信号清单

---

有个问题我被问过很多次。

“Seven，我该用 Lovable 还是 Cursor？Claude Code 是不是更厉害？要不要直接上最复杂的？”

我的回答一直很统一：选最适合你现在这个阶段的，不是最贵或最复杂的。

工具不是武器。不是越贵、越复杂就越好。工具更像鞋：爬山穿登山靴，去海边穿拖鞋，上班穿皮鞋。你不会穿着登山靴去海边，虽然登山靴确实更“强”。

我做了五个项目，每个项目用的工具不同。sevenailab 用 Lovable 两天做完。伴聊用 Cursor。Say it 和 7brandhub 用了 Codex。ClawChat 用了 Claude Code。每次换工具都是因为项目需求变了，跟“升级”没关系。

这一章帮你理清这几类工具的定位，以及什么时候该从一个换到另一个。

---

## 工具三级梯度

我把 AI 编程工具分成几个层级，每一级适合不同的项目需求和阶段。

```text
【工具梯度】

第一级：Lovable（全托管型）
  你说需求，它做一切。
  不碰代码，不碰终端，不碰部署。
  适合：第一个产品、官网、落地页、快速验证想法。

第二级：Cursor / Codex（协作型）
  Cursor：你在本地的编辑器里看到代码，AI 帮你写和改。
  你能看到每一步发生了什么，也能自己调。
  
  Codex：OpenAI 出的 AI 编程工具。
  你可以在网页里委托它改代码、跑测试，
  也可以把它接到终端或 IDE 里配合本地项目使用。
  这本书里主要把它当成云端编程助手来理解。
  适合：需要接 API、改逻辑、做复杂交互的产品。

第三级：Claude Code（命令行型）
  在终端里对话，AI 理解你整个代码库。
  能跨文件编辑、运行命令、并行处理多个任务。
  适合：大项目、多人协作、需要精确控制的场景。
```

这几个层级不是线性的升级。有些人用 Lovable 做完了收入不错的 SaaS（Software as a Service，在线软件服务，通常按月或按年订阅使用），从来没碰过 Cursor。有些人一开始就用 Claude Code，因为他们的项目天然需要终端操作。Cursor 和 Codex 在这一章里放在同一层理解：它们都更接近“你能看到代码、能审改动、能接入真实工程流程”的协作型工具。选哪个看你的习惯和当时可用的入口：喜欢本地编辑器用 Cursor，喜欢把明确任务交给 OpenAI 的编程助手处理，就关注 Codex 的最新形态。




关键看你的项目需要什么，跟你“在哪一级”没关系。

## 市面上不止这三个

AI 编程工具的赛道很热，新工具隔几周就冒一个出来。你大概率已经听过不少名字了。这里做一个最简单的分类，帮你建立坐标感：

```text
类型          代表工具              一句话定位
全托管型      Lovable / Bolt        说需求，出产品，不碰代码
协作型        Cursor / Windsurf     你看代码，AI 帮你写和改
云端代理型    Codex / AI Studio Apps 等    云端或工作区里执行任务，改完交你审
命令行型      Claude Code           终端对话，理解整个代码库
设计转代码    v0.dev                输入描述，生成 UI 组件
```

你不需要每个都试。这张表的作用是：当你在社交媒体上看到有人安利某个工具时，你能快速判断它跟你现在用的是同一类还是不同类。同一类的工具之间差别没你想象的大；不同类的工具解决的是不同阶段的问题。

我选 Lovable 而不是 Bolt 做第一个项目时，没有做过严格对比。决定因素很朴素：Lovable 那时候用户增长最快、社区活跃度最高、踩坑帖最多——踩坑帖多意味着你遇到问题时容易搜到解决办法。选工具和选产品一样，有人用比功能多更重要。

如果你现在用 Lovable 用得顺，不需要换。如果你已经在用 Cursor，也不需要换。先把手上的项目做完，再看要不要调整。下面我把每一级工具展开讲讲。

---

## 第一级：Lovable

Lovable 是一个全栈 AI 开发平台。你用自然语言描述需求，它生成包含前端、后端、数据库、认证的完整应用。

Lovable 官方对自己的定义是：一个全栈 AI 团队，不只是“建站工具”。预览、分析、云、代码、安全、速度，这些功能面板覆盖了前端、后端、运维、安全、数据分析的常见角色。

它适合什么？官网、落地页、SaaS 原型、内容平台、内部工具、客户仪表板。从 MVP 验证到生产级应用，它都能处理。

它对谁设计？四类人：个人创业者（快速 MVP 验证）、产品和市场团队（原型和协作）、技术团队（代码可移植性）、企业（安全合规）。

它最打动我的地方，是起点高。你第一次把需求文档贴进去，它生成的不是一个空白模板，而是一个 80 分的完整产品。你可以直接在上面精修，省掉“搭脚手架”的阶段，直接进入调方向、改细节。

我做 sevenailab 的时候选 Lovable，外部信号也帮了忙：它在短时间内获得了很高的增长和大量真实用户。市场已经替你验证过一部分。挑工具时看增长趋势、社区活跃度和踩坑帖，比只看功能列表更能帮新手避雷。

你到目前为止用的就是 Lovable。它帮你走完了“定生修验发察”的完整循环。

但它有边界。

---

## Lovable 的边界在哪

Lovable 像一个“给你做菜的厨师”。你点菜，它做。但你看不到厨房里发生了什么。

等你的需求变得更复杂，需要接入第三方 API、处理复杂的数据流、自定义交互逻辑，你会越来越频繁地发现：“我想做的事，在 Lovable 里描述不清楚。”

还有一个更现实的问题：代码不在你手里。

Lovable 的代码存在它的云端。虽然它支持连接 GitHub、导出代码，但你的日常开发流程是在它的网页界面里完成的。如果有一天你需要换工具、需要多人协作、需要精确控制部署流程，你得先把代码拉出来。

什么时候该考虑升级？我给你一个清单：

```text
【升级信号清单】

从 Lovable 升级到 Cursor / Codex 的信号：
□ 你发现自己在 Lovable 里反复描述同一个修改但它改不对
□ 你需要接入外部 API 且调试过程很痛苦
□ 你想看到代码结构、理解哪些文件在做什么
□ 你的产品有超过 10 个页面或复杂的状态管理
□ 你需要使用 Lovable 不支持的技术或库

从 Cursor / Codex 升级到 Claude Code 的信号：
□ 你的项目已经有几十个文件，跨文件修改越来越频繁
□ 你需要运行脚本、批量处理、自动化操作
□ 你想让 AI 理解整个代码库而不是单个文件
□ 你需要多个 AI 并行处理不同的任务
□ 你的团队多人协作，需要精确的权限控制
```

碰到一个信号不一定要立刻升级。但如果清单上的勾越来越多，就是时候了。

## 第二级：Cursor

Cursor 是一个 AI 加持的代码编辑器。

如果 Lovable 是“给你做菜的厨师”，Cursor 就是“和你一起做菜的搭档”。你能看到厨房里的一切：食材（代码文件）、操作台（编辑器）、火候（运行状态）。AI 在旁边帮你切菜、调味，你随时能伸手调整。

Cursor 有四个核心区域：侧边栏（文件管理、搜索、Git）、编辑区（看和写代码）、终端（运行命令）、AI 聊天面板（和 AI 对话）。

这里要解释几个你会碰到的新概念。

**IDE** 是什么？就是“代码编辑器”的正式名字，集成开发环境（Integrated Development Environment）。Cursor 就是一个 IDE。你可以把它理解成“写代码专用的 Word”。

**终端（Terminal）** 是什么？就是一个输入文字命令的窗口。你在里面敲一行命令，电脑就执行一个动作。比如 `npm run dev` 是让项目在本地跑起来，`npm run build` 是检查代码能不能编译通过。

终端看起来很吓人，一个黑底白字的窗口，没有按钮，没有图标。但你实际用到的命令很少。我做了五个项目，日常就用这几个：

```text
cd my-project     → 进入项目文件夹
npm install       → 安装项目需要的依赖
npm run dev       → 在本地启动项目
npm run build     → 检查代码能不能编译
```

四个命令，够用了。不需要更多。

**npm** 是什么？全称 Node Package Manager。你可以把它理解成“代码世界的 App Store”。别人写好的工具和库，你用 `npm install` 一键安装就能用。

Cursor 的三大核心操作：让它理解现有代码、在本机运行代码（`npm run dev`）、重构代码结构。完成后添加实际功能，如 API 集成。

Cursor 还有一个特别好用的功能叫 Checkpoint，会在重大改动前自动创建快照。你可以随时预览之前的状态、选择恢复到某个版本，或者在那个版本基础上分支尝试。这让探索性开发变得低风险可回滚。不用怕改坏了回不去。

还有一个插件叫 Stagewise，实现“哪里不爽点哪里”。在浏览器里直接点击元素，Cursor 自动定位到对应的代码并修改。操作很简单：打开 Cursor 搜索 stagewise，安装，运行项目，然后在 localhost:3000 上点你想改的地方。

## 第三级：Claude Code

Claude Code 是一个在终端里运行的 AI 编程工具。

没有图形界面。没有预览窗口。你打开终端，输入 `claude`，然后用自然语言和它对话。

听起来更原始，但它对代码库级任务很强。

为什么？因为它能在终端环境里读取项目文件、理解多个文件之间的关系，而不只是看单个文件。它能跨文件编辑、运行命令、集成开发工具。

官方的入门流程只有五步：安装 → 登录 → 启动 Session（cd 到项目目录，运行 `claude`）→ 提问（让它分析代码库）→ 改代码（描述改动，审批后执行）。整个流程充满对话感，你和 AI 来回沟通、逐步推进。

Claude Code 也支持更进阶的子代理（subagents）工作方式：你可以为代码审查、测试、文档等任务配置专门角色，让它们在合适场景里分工处理。对于大项目里的并行审查、跨文件重构、复杂集成，这类能力很有价值。但这是进阶用法，不是第一个产品必须掌握的东西。

我做 ClawChat 的时候用的就是 Claude Code。那个项目需要两个 AI 角色之间实时对话、流式输出、状态管理。这些需求在 Lovable 里很难描述清楚，在 Cursor 里也需要频繁手动协调。但在 Claude Code 里，我可以让它直接理解整个项目的逻辑，然后一次性修改涉及的多个文件。

但 Claude Code 也有明确的门槛：它纯命令行操作、API 费用比较高、需要你对代码结构有基本理解。如果你还在做第一个产品，不建议从这里开始。

## 我做五个项目的真实工具选择

把我的五个项目摊开看，你就能看到工具选择的逻辑：

**sevenailab**（GEO 服务官网）→ Lovable。需求简单明确：官网 + 预约入口。两天做完。不需要复杂逻辑。

**伴聊**（微信小程序）→ Cursor。需要接微信 API。API 就是“借用别人的功能”：你的产品需要 AI 回复，就调用大模型的 API；需要发邮件，就调用邮件服务的 API。它还要处理对话流、管理用户状态，Lovable 搞不定这些。

**Say it**（内容工作流）→ Codex。AI 采访 + 多格式内容生成，涉及 API 调用和复杂的数据处理。Codex 的优势是可以接管一段明确任务，读代码、改代码、跑测试，再把结果交给你审核。具体入口和产品形态会变化，以 OpenAI 官方说明为准。

**7brandhub**（品牌服务商平台）→ Codex。平台型产品，有数据库、搜索、分类。连上代码仓库后，用自然语言描述改动，Codex 执行修改，改完再由你审批。

**ClawChat**（Agent Demo）→ Claude Code。实时对话、多角色、流式输出。需要 AI 理解整个代码库的上下文，跨文件协调修改。

你看到规律了吗？

项目的需求决定了工具的选择。跟我个人“变厉害了”没什么关系。

简单的官网，用 Lovable 最快。需要接 API 的产品，用 Cursor 更可控。需要跨文件大改的项目，Claude Code 最合适。

## 一个常见错误：工具焦虑

AI 编程工具更新很快。每隔几周就有一个新工具出来：Bolt、v0.dev、AI Studio Apps、AugmentCode、Windsurf。每个都说自己是“最好的”。

不要陷入工具焦虑。

我初期在 Lovable、Cursor、Bolt 之间跳来跳去，两个月下来什么都没做成。后来我逼自己只用 Cursor 做完伴聊，三周出了成品。工具焦虑的解药就一个字：定。选定一个，用到底。

AI 编程工具的学习路径建议：先用 Lovable（或 Bolt）快速做原型，再用 Cursor 本地打磨，最后根据需要选择更高级工具。没有绝对最好的工具，只有最适合的场景。

我在“回看”那一章说过一句话：停止追新功能、停止收藏教程。就盯着手头这个项目，一个一个问题地解决。工具选择也是一样。先把一个用到位，再考虑换不换。

## 用开源项目加速：Cursor 的隐藏玩法

当你开始用 Cursor，有一个很多人不知道的加速方式：用现有的开源项目当起点。

流程是这样的：找到差异化的用户人群和痛点 → 寻找能帮你解决问题的开源项目 → 用 Cursor 修改核心功能 → 修改界面和交互 → 发布。

第一步还是你自己的判断：目标用户是谁，痛点是什么。开源项目给你的是一个起跑线，但往哪个方向跑，不能交给模板决定。

这个方法的好处是：直接在一个已经能跑的项目上做定制，不用从零搭框架。结合 Cursor 的 AI 能力，修改和扩展很快。

## 工作流三层模型

做了五个项目后，我总结出一个个体做产品的工作流模型：

第一层：我不一定亲手写代码，但我必须掌握需求和判断权。这是你作为“大脑”的角色。

第二层：让一个 AI 负责具体代码执行，另一个 AI 去 review 方案。比如我用 Claude Code 写代码，用 Claude 对话版 review 思路。

第三层：选择合适的工具匹配项目阶段。验证期用 Lovable 快速出原型，打磨期用 Cursor 或 Codex 精细控制，规模期用 Claude Code 高效协调。

这三层不是“读完一层升级下一层”。它们是同时存在的。即使你现在只用 Lovable，第一层的意识也一直在起作用：掌握需求和判断权。

---

## 从 MVP 到完整产品还差什么

最后说一个你迟早会碰到的问题。

你现在做的是 MVP，最小可行产品。它能跑、能用、能给人看。

但从 MVP 到一个完整的 SaaS 产品，中间还有好几层东西：用户系统（注册、登录、账户管理、权限控制）、变现方式（定价策略、会员层级、功能限制）、付费体系（支付网关、订阅管理、发票）、国际化支持（多语言、多货币）。

这些东西不需要现在做。但你需要知道它们存在。

因为当你的产品从“给朋友试用”变成“真的有人付费”的时候，你会需要逐步加上这些能力。到那时候，你大概率已经不在 Lovable 里了，你需要 Cursor 或 Claude Code 来处理这些复杂度。

工具梯度的本质就是这个：在对的时候用对的工具。同样的原则放在产品能力上也成立：知道怎么借用现成的能力，不必什么都自己写。

## API 积木卡

从“网页”走向“AI 产品”，有一个关键的思维转变：把产品拆成 API 积木。

API 可以理解为“借用别人已经做好的能力”。你要让产品生成文案，就调用大模型 API；你要语音转文字，就调用 Whisper 之类的语音 API；你要发邮件，就调用邮件服务 API；你要收款，就调用支付 API。

很多厉害的产品，靠的是把几个现成 API 用很聪明的方式串起来，不是什么全新技术。

你可以用这张卡拆你的产品：

```text
【API 积木卡】

1. 我的产品需要借用什么能力？
   → 文本生成 / 图片生成 / 语音识别 / 支付 / 邮件 / 地图 / 数据库

2. 这个能力有没有现成 API？
   → 硅基流动 / OpenRouter / fal.ai / Replicate / Stripe / 微信支付 / Resend

3. 官方文档在哪里？
   → 把文档链接发给 AI，让它先读文档再动手

4. URL 和 Key 在哪里拿？
   → 从服务商后台获取
   → public key 可以给前端用；secret key 只能放在 `.env.local`、服务端或部署平台环境变量里

5. 最小测试请求是什么？
   → 先让 AI 写一个最小 Demo，确认能调用成功

6. 成功和失败怎么验收？
   → 成功时看到什么结果；失败时用户看到什么提示
```

这张卡不用每次都填很完整。它的作用是让你意识到，所谓复杂产品，往往不是“我要学一堆代码”，而是“我要搞清楚每块外部能力怎么接进来”。AI 可以帮你写接入代码，但你要知道自己在接什么，为什么接，接完怎么算成功。

## 要点小结

工具选择的原则很简单：选最适合当前项目的，不是选最复杂的。深入掌握一个工具 > 浅尝多个工具。工具焦虑的解药是“选定一个然后用到底”，而不是频繁跳来跳去。

**本章行动**：用本章的升级信号清单评估你的项目，确定该继续用 Lovable，还是升级到 Cursor / Codex / Claude Code。接下来的一个月里，先把最适合当前项目的那一个工具用透。
