# 第十九章 · 实战 2：一周做出一个微信小程序

> 所属部分：第五部分 · 实战，四章五个真实案例  
> 本章路线：国内路线（微信生态、微信支付、小程序备案）  
> 本章方法：微信小程序上线清单

---

sevenailab 是海外路线：英文官网、Lovable 一键部署、域名在 Namecheap。

伴聊是国内路线。微信生态，完全不同的规则。

这一章用伴聊的案例，复盘一个微信小程序从调研到上线的完整过程。如果你想做面向国内用户的产品，这一章比上一章更重要。




---

## 背景：为什么做伴聊

起点是一个很小的痛点：收到一条消息，不知道怎么回。

不是不想回。是拿捏不准分寸。职场上怎么回不失礼？感情里怎么回不尬？朋友问了一个敏感问题怎么圆过去？

我发现身边很多人有这个问题。于是就想做一个极简的工具：你把对方的消息贴进来，AI 帮你生成 3 种风格的回复建议。

这就是伴聊。极简聚焦，只做一件事：帮你回消息。

```text
【案例复盘卡：伴聊小程序】

起点：用户收到消息后，不知道怎么回、怕回错语气
关键动作：先验证需求，再把核心闭环砍到输入消息 → 选择场景 → 复制回复
最大坑：微信小程序的难点不只在代码，更在类目、备案、支付和审核
关键判断：模型质量决定产品体验，便宜模型不一定适合中文情绪场景
可复用动作：国内产品先检查微信生态规则，再决定小程序还是 Web
```

---

## 工具栈选择

伴聊选了 Cursor 作为主开发工具，而不是 Lovable。

为什么？因为微信小程序有自己的技术规范，WXML（微信 XML）和 WXSS（样式文件），运行在微信的运行环境里，不是标准的 Web 页面。Lovable 生成的是标准 Web 应用，不能直接跑在微信里。

工具栈是：Cursor（AI 辅助编程）+ 微信开发者工具（小程序官方 IDE）+ Claude（辅助思考和生成文档）。

---

## 从调研到 PRD

在写代码之前，我先做了两件事。

按第四章讲的方法，我先让 AI 帮我做了一轮市场验证，确认“回复建议”这个方向有真实需求，然后按第五章的方法用 Claude 生成了 PRD。伴聊的不同之处在于它足够极简，只有一个核心闭环里的三个动作：输入消息、选择场景、复制回复，没有任何“以后可能用到”的东西。

准备工作做得越充分，后面开发越顺利。急着写代码的结果往往是写到一半发现逻辑不对，又要推倒重来。

---

## AI 模型选型

伴聊的核心是“帮你回消息”。AI 回复的质量，直接决定产品能不能用。

一开始我用智谱和 DeepSeek，因为便宜。但测试发现输出太死板、太“AI 味”。

后来换了幻宙（专门为中文情感对话训练的模型）。测试同一个场景，用户输入“在吗？”，智谱回复“你好，有什么可以帮您？”，幻宙回复“在呀，咋啦？”。

差别一目了然。

教训：不要盲目跟风便宜方案。测试真实场景才能验证质量。模型选型要看它在你的具体场景里够不够自然，参数排名说了不算。

---

## Cursor Rules 和开发

开发之前，我先让 Claude 帮我生成了一份 Cursor Rules。这个方法在第十二章已经讲过：先把项目规范写成长期规则，后面 Cursor 执行时就不容易跑偏。

“我要用 Cursor 开发微信小程序，请帮我生成一份 Cursor Rules 文件。要求包括代码风格规范、小程序开发最佳实践、错误处理原则、代码审查检查项、微信小程序的特殊要求。”

有了这份 Rules，Cursor 写出来的代码从一开始就很规范。

然后开始写功能。核心流程很短：用户输入消息 → 选择场景（职场/情感/社交）→ AI 生成 3 种回复 → 用户复制使用。

整个过程的 Bug 修复方式和之前讲的一样：把所有看到的问题列成清单，让 Cursor 一次性修复。不是一个 Bug 修完再修下一个。

## 微信小程序上线的坑

代码写完了，但上线远比你想的复杂。

微信小程序不像 Web 产品，写完代码推到 Vercel 就完事。它有一套自己的审核和备案流程。

```text
【微信小程序上线清单】

1. 微信公众平台注册
   → 需要选择主体类型并完成认证
   → 认证费用和优惠政策会变化，以微信公众平台当前说明为准

2. 类目审核
   → 涉及 AI 生成内容时，类目和资质要求会更严格
   → 可能需要提供大模型服务商材料、算法或 AI 生成内容相关说明、订单证明等
   → 当时路径：我通过微信服务市场获取了 MiniMax 的订单材料（当时免费额度也有订单）

3. 小程序备案
   → 通过微信认证后进行备案申请
   → 审核时间取决于资料完整度和平台口径
   → 不完成备案，发布和搜索能力可能受影响

4. 微信支付商户号
   → 需要公司或个体工商户主体
   → 配置支付参数（商户号、回调地址、环境变量）

5. 提交审核
   → 微信官方审核你的小程序内容
   → 不符合规范会被拒（可修改后重新提交）
```

我第一次提交就被拒了。原因是申请的“效率-工具”类目不符，平台认为这个产品涉及 AI 生成内容，需要按当时的 AI 相关类目和材料重新补充。




追加类目需要提供大模型服务商的相关材料。我当时通过微信服务市场获取了 MiniMax 的订单材料，把订单截图提交后审核通过。

这是当时可行的曲线方案，不代表永远可行。MiniMax 的免费额度、大模型服务商材料要求、微信对 AI 类小程序的审核口径都会变化。你提交前一定要去微信公众平台看最新类目和资质说明，别只照抄我的路径。

还有一个容易低估的前置动作：备案。小程序备案通常要准备主体证件、负责人信息、联系方式、服务内容说明等材料。审核时间可能是 1-3 天，也可能因为资料补正拉长。只要你确定要做国内小程序，备案和类目审核最好第一天就开始看，不要等代码写完才发现上线还要排队。

## 定价

伴聊的 MVP 定价策略：

免费额度：新用户 3 次免费体验。让用户先试，不设门槛。

周订阅：20 次对话，6.9 元。门槛低，让用户试用后决定要不要付费。

月订阅：60 次对话，19.9 元。性价比更高，给长期用户一个理由。

定价的核心思路是：第一版不追求利润最大化，追求有人愿意付费。只要有人付了钱，就说明产品有真实价值。

---

## 支付接入

支付是整个流程里最烦的一步。

商户号、回调地址、环境变量、支付链路、前后端联调，每一步都比你想象中多一层坑。

但这一步不能跳过。最好一开始就想清楚最后要往哪接。不然很容易做着做着，产品跑通了，但收钱这件事完全没设计过。

国内产品用微信支付最自然。你的用户就在微信里，用微信支付最顺手。不要假装大家都接受 Stripe。

---

## 伴聊的差异化

最后说一个做产品的思路。

伴聊不是“套壳 GPT”。它的差异化在三个地方：

第一，极简聚焦。只做一件事：帮你回消息。不做“AI 助手”“万能聊天”。

第二，定制化 prompt。系统针对中文沟通场景深度定制了 System prompt，不是直接把用户消息扔给 AI，而是先识别场景类型（职场/情感/社交），再根据场景生成不同风格的回复。

第三，用户体验设计。三种回复风格让用户有选择感。复制功能一键到位。界面极简到打开就会用。

差异化靠的是在一个极小的点上做得比别人好。功能堆得再多也没用。

---

## 如果卡住了

小程序开发最容易卡在审核环节。被拒了不要慌，仔细看拒绝理由。最常见的两种：一是类目选错了（比如你做的是“工具”但选了“社交”），二是缺少必要的资质文件（比如涉及 AI 内容生成需要额外资质）。

类目被拒的解决办法：去微信公众平台查最新的类目列表，找到最匹配你产品实际功能的类目，重新提交。如果不确定，选范围更大的类目比选范围更小的安全。

另一个卡点是支付商户号的申请。这个流程比你预期的长，通常需要 3-7 个工作日，遇到补充材料还会更久。如果你的产品需要收费，第一天就把商户号申请路径查清楚，不要等到产品做完了再处理。

---

## 要点小结

微信小程序和 Web 产品是两条完全不同的路线，技术栈不同、上线流程不同、用户触达方式不同。最大的区别在微信的审核规则，而不是代码本身。类目、备案、支付商户号都是绕不过去的。伴聊从调研到上线用了一周，被类目审核拒过一次。这次被拒反而澄清了要走哪条路。重点是快速理解规则、调整、再提交，被拒本身不是问题。

这一章讲的是我的个人经历，不是微信小程序合规指南。类目、材料、备案、认证费用、支付审核都会变化，真正提交前一定要以微信公众平台和微信支付后台的最新要求为准。

**本章行动**：确定你的产品是否面向国内用户。如果是，去微信公众平台注册一个测试小程序账号，提前了解认证流程和类目选择，而不是等到上线才处理。选错类目意味着后面被拒，浪费时间。
