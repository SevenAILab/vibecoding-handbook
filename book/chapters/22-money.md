# 第二十二章 · 收钱：第一笔钱从哪里来

> 所属部分：第六部分 · 闭环，从产品到生意  
> 读完本章，你会得到：支付接入清单 + 第一版定价方案  
> 本章方法：定价最小决策 + 支付双轨对照

---

做了这么多章，有一个问题你可能一直在回避。

“这个东西怎么收钱？”

很多人想收钱，但卡在三个问题上：收多少、怎么收、什么时候收。

我做伴聊的时候也有过这个纠结。产品刚做出来，还有 Bug，功能还不完善。这时候收钱？会不会太早？会不会得罪用户？

后来我想通了一件事：收钱不是“产品完美了才做的事”。收钱是验证产品价值最直接的方式。

有人愿意付费 = 你做的东西至少产生了强烈兴趣，甚至已经有真实价值。

没人愿意付费 = 你需要重新想想方向。没人付费也可能是定价太高、支付流程太麻烦、或者你的用户根本不知道这个产品存在。不一定是产品本身没价值。但付费是很强的信号。

免费用户的反馈再好，也不如一个付费用户的反馈有说服力。

但第一笔钱也不是终点。

第一笔钱证明用户愿意试一次。第二笔钱、续费、主动推荐，才证明产品真的进入了他的工作流。




很多产品的第一笔钱验证的是“好奇心”和“尝鲜”。更有分量的验证，是用户下个月还愿意回来。

## 定价最小决策

第一版定价不需要完美。你只需要做三个决定：

```text
【定价最小决策】

1. 免费还是付费？
   → 推荐：有免费试用 + 付费升级
   → 让用户先体验价值，再决定付费

2. 按次还是按月？
   → 按次：适合低频工具（用一次付一次）
   → 按月：适合高频产品（每月订阅）
   → 不确定就先按月，月价更容易定

3. 多少钱？
   → 国内：9.9 / 19.9 / 49.9 元/月
   → 海外：$5 / $9 / $19/月
   → 第一版选最低档就行，后面可以调

4. 成本底线是多少？
   → 如果你的产品调用 AI 模型，先估算一次使用要花多少钱
   → 单次成本 × 用户可用次数，不能高过你的收费
   → 不用算得像财务模型，但至少别卖一单亏一单
```

伴聊的定价就是按这个思路来的：免费 3 次体验 + 周订阅 6.9 元 + 月订阅 19.9 元。




第一版定价先别想着利润最大化。先确认有人愿意付费，同时保证成本结构别明显倒挂。

## 支付接入：国内 vs 海外

接支付是很多人最怕的一步。确实有点烦，但它不是天书。

如果你面向国内用户，用微信支付。如果面向海外用户，用 Stripe。下面是两条路线的具体操作步骤。




```text
【海外路线：接入 Stripe 的具体步骤】

准备阶段（提前 1-2 周）：
  1. 确认你能用哪种主体注册 Stripe
     → Stripe 目前不直接支持中国大陆个人注册
     → 常见路径包括香港公司、美国公司、新加坡公司、Stripe Atlas 或合作伙伴平台
     → 具体国家/地区、材料和费用以 Stripe 官网最新政策为准
  2. 准备对应主体的银行账户或收款账户
     → 例如香港公司可考虑 Airwallex 等线上账户
     → Stripe 收到的钱会打到这个账户

接入阶段：
  1. 打开 stripe.com，用公司信息注册账号
  2. 完成身份验证（KYC），上传公司文件和身份证明
  3. 在 Stripe 后台找到 API keys（通常在 Developers → API keys 一类入口）
     → 你会看到两个 Key：Publishable key 和 Secret key
     → Publishable key 可以给前端用，Secret key 只能放在服务端
     → 本地开发时放进 `.env.local`，上线后放进部署平台环境变量
  4. 告诉 AI："帮我接入 Stripe 支付。
     我要做 [一次性支付 / 订阅付费]，
     价格是 $X/月。Stripe 的 key 在 `.env.local` 或部署平台环境变量里。
     请使用 Stripe Webhook，并验证 webhook signature。"
     → AI 会帮你创建支付页面和回调处理
  5. 用 Stripe 的测试卡号（4242 4242 4242 4242）走一遍支付流程
     → 确认能正常扣款、跳转、显示成功页面
     → 更重要的是确认 Webhook 收到成功事件，并且数据库里的用户权限被更新
  6. 测试没问题后，在 Stripe 后台切换到"正式模式（Live mode）"
```

```text
【国内路线：接入微信支付的具体步骤】

准备阶段（提前 1-2 周）：
  1. 你需要一个公司或个体工商户的营业执照
     → 个体户也行，去当地市场监管局办，一周左右
  2. 在 pay.weixin.qq.com 申请微信支付商户号
     → 用营业执照信息注册，等审核（通常 3-7 个工作日）
     → 具体类目、费率、资料要求以微信支付后台为准
  3. 提交资料
     → 通常包括营业执照、法人/经营者身份证、结算银行账户、客服电话、经营类目说明
     → 审核中可能会要求补充材料，预留几天缓冲
  4. 审核通过后，在商户后台找到：商户号、API 密钥、证书

接入阶段：
  1. 把商户号、API 密钥和证书相关配置存到 `.env.local` 或部署平台环境变量里
  2. 告诉 AI："帮我接入微信支付。
     商户号和密钥在 `.env.local` 或部署平台环境变量里。
     我要做 [JSAPI 支付（小程序内）/ Native 支付（扫码）]。
     请按微信支付当前 API 文档处理回调通知、签名验证和订单状态更新。"
  3. AI 会帮你写支付代码和回调接口
  4. 用微信支付的沙箱环境测试（或小金额实测）
  5. 确认收款后在商户后台能看到订单，并确认你的数据库订单状态已更新
```

三个提醒：第一，不管哪条路线，支付相关的 secret key、API 密钥和证书一定存 `.env.local` 或部署平台环境变量里，不要写在代码中。第二，支付成功不能只看前端跳转页面，必须以后端回调或支付平台通知为准，并做签名验证。第三，商户号、公司注册、类目审核、KYC/KYB 这些准备工作都需要时间，第一天就把申请提交了，不要等到产品做完了才开始办。

还有一个现实判断：如果你还没有任何用户，不一定要第一天就把支付代码接完。但你必须第一天就想清楚支付路径。

“后接支付”和“后想支付”是两回事。前者是合理排序，后者是埋坑。

### 国内收钱的现实

上面的微信支付流程看起来很清晰，但实际操作时你大概率会卡在第一步：营业执照。

微信支付要求你有公司或个体工商户主体。如果你只是一个人在做产品，这意味着你要先去注册一个个体户，然后提交审核，等几个工作日。对很多人来说，“注册个体户”这件事本身就够劝退了。

但先别因为这个放弃。国内个人开发者收到第一笔钱，不一定非要走微信支付正式接入。

几个替代路径：微信小商店可以卖虚拟商品，个人就能开，审核比微信支付简单。知识星球或小报童可以卖内容或服务包，不需要任何开发。如果你的产品是帮人做一件具体的事（比如帮人改简历、帮人做 GEO 诊断），最简单的方式就是先手动交付：用户微信转账，你手动发结果。

这些都不优雅。但第一笔钱的意义不在于流程多漂亮，而在于有人真的为你做的事掏了钱。

我做 sevenailab 的 GEO 诊断服务时，第一笔钱就是客户微信转账的。没有支付页面、没有订单系统、没有自动化。就是聊了需求，我做了诊断报告，他觉得有价值，转了钱。后来有了更多客户，才开始考虑标准化和正式支付接入。

所以如果你现在被“微信支付接入”卡住了，先别卡在这里。先用最土的方式收到第一笔钱。等你验证了有人愿意付费，再花时间去搞正式的支付流程，那时候你的动力会完全不一样。

## 海外路线的优势

如果你考虑做面向海外的产品，有几个结构性优势值得知道。

同一个效率工具，国内卖 19.9 元/月，海外定 $9/月，收入差不多差 3 倍，开发成本却差不多。这就是跨市场套利的空间。

我不是说一定要做海外。但如果你的产品天然适合英文市场（比如开发者工具、设计资源、效率工具），海外路线值得认真考虑。

---

## SaaS 网站必需的法律页面

开始收钱之后，有几个法律页面必须有：服务条款（权利义务）、隐私政策（数据保护）、退款政策（购买保证）。

这三个必须有。初期可以用 AI 生成法律条款，后期根据实际业务调整。但如果涉及真实的法律合规，建议找专业人士确认。

## 要点小结

收钱是验证产品价值最直接的方式，但第一笔钱不是最终答案。第一版定价不追求利润最大化，先回答四个问题：免费还是付费模式、按次还是按月计费、多少钱、成本底线是多少。国内用微信支付，海外用 Stripe。支付接入不只是配置几个参数，还包括主体资质、账户审核、密钥安全、回调验证和法律页面。提前想清楚路径，别等到产品做完了才发现“怎么收钱”是个大坑。

**本章行动**：拿出你的产品，用“定价最小决策”四问写出第一版定价方案。然后确定你走国内还是海外路线。如果还不确定，至少搞清楚你的目标用户在哪里、他们用什么支付方式，以及你的单次服务成本大概是多少。这归根到底是商业问题，技术反而是最后才操心的。
