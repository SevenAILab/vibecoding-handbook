# 第七章 · 修：别让 AI 猜，给它证据

> 所属部分：第二部分 · 主控流程，定生修验发察  
> 读完本章，你会得到：修改记录 + 一份“修改 prompt 模板”  
> 本章方法：修改 prompt 三段式 + 五类报错分层

---

第一版跑起来了。恭喜。

但我几乎可以确定：它有不对的地方。

按钮颜色不对。文字太大或太小。某个流程走到一半断了。有一个地方你看着总觉得别扭但说不上来哪里不对。

这些都正常。

这一章要解决的问题是：怎么让 AI 帮你改？

答案只有一个：精确地告诉 AI 哪里不对、什么样算对。

---

## 修改的质量取决于你描述问题的质量

我做 sevenailab 的时候，全程用自然语言改设计。这是当时的一些指令：

“把英雄界面板块的背景换成弥散的渐变蓝。”

“帮我上传这张图，放到‘服务流程’板块。”

“把这个客户评价做成轮播的卡片样式。”（附参考：苹果官网的动效）

全程没碰过 CSS，响应式自动适配。

这些指令之所以有效，因为每一条里都有三样东西：具体的目标模块（“英雄界面板块”“服务流程板块”“客户评价”）、具体的视觉或交互动作（“换背景”“放图片”“做成轮播”）、一个可参考的审美锚点（“弥散渐变蓝”“苹果官网动效”）。

反过来看，最没用的修改指令是什么？

“帮我把这个页面改好看一点。”

“这里不太对，帮我调一下。”

“整体感觉不够高级。”

这些话你说了等于没说。AI 只能猜，猜完你再说“不是这样”，再猜，再不对。三轮下来，你的耐心没了，AI 的上下文也乱了。

所以我给你一个修改 prompt 的模板，我叫它“修改 prompt 三段式”：

```text
【位置】：____（哪个页面 / 哪个模块 / 哪个按钮）
【现象】：____（现在是什么样 / 发生了什么问题）
【期望】：____（我希望它变成什么样 / 参考什么）
```

比如：

“首页的 CTA 按钮（位置），现在是灰色、字很小、不显眼（现象），我希望它变成蓝色圆角按钮，字号加大，鼠标悬停时颜色加深（期望）。”

“服务介绍页第二个卡片（位置），点击后没有任何反应（现象），我希望点击后跳转到详情页或弹出一个介绍弹窗（期望）。”

位置 + 现象 + 期望。三段。每一段都有具体信息。

## 不要一个 Bug 一个 Bug 改

上一章我提过一句：不要一个 Bug 修完再修下一个。

这里展开说。

很多人的习惯是：发现一个问题，发给 AI，等它改完，再去看下一个问题，再发，再改。

这样做有两个问题。

第一，效率低。每一轮修改 AI 都要重新读取上下文、重新理解代码、重新生成。一个一个修，等于让它重复这个过程十遍。

第二，容易互相冲突。你改了按钮颜色，AI 顺手动了一段 CSS；你再改布局，它又动了同一段 CSS。改到后面，某个之前修好的地方又坏了。

更好的做法是：先走一遍完整的用户流程，把所有你看到的问题列成一个清单，然后一次性交给 AI。

我做 Say it 的时候就是这样。第一版生成后，我从“打开页面 → 开始对话 → 完成对话 → 生成内容”走了一遍，记下所有不对的地方，然后一次性提交：

```text
请修改以下问题：

1. 首页标题文字太大，改成 36px
2. 导航栏在手机上会被截断，需要做响应式适配
3. "开始对话"按钮点击后没有反应，应该跳转到对话页
4. 生成内容的三个版本（小红书/公众号/朋友圈）间距太窄，增加间距
5. 页面底部的版权信息年份不对，改成 2026
```

一次性提交，AI 一次性改完。然后你再走一遍流程，看有没有遗漏。

这个“列清单 → 一次性修”的节奏，比“发现一个改一个”快三倍以上。

## 给 AI 一个上下文包

如果你的产品开始变复杂，光说“这里坏了”就不够了。

AI 很聪明，但它不知道你手上发生了什么。你看到的是完整现场：页面长什么样、你点了哪里、之前改过什么、你希望变成什么。但 AI 只看到你发过去的那几句话。你不给上下文，它就只能猜。

所以从这一章开始，遇到复杂修改或 Bug，我建议你打包一个“上下文包”。




```text
【上下文包】

1. 当前目标
   → 我现在要修什么 / 做什么功能

2. 当前状态
   → 页面现在是什么样，最好附截图或 URL

3. 操作步骤
   → 我点了哪里，输入了什么，然后发生了什么

4. 期望结果
   → 正常情况下应该发生什么

5. 证据材料
   → 报错文字、Console 截图、终端日志、Vercel Logs、设备浏览器信息

6. 不要改什么
   → 哪些功能现在是好的，不要顺手重构，不要改设计风格
```

比如：

```text
【当前目标】修复预约表单提交失败的问题。
【当前状态】本地能提交，部署到 Vercel 后提交失败。页面 URL 是 ____。
【操作步骤】打开首页 → 填写姓名和邮箱 → 点击"预约诊断"。
【期望结果】提交后显示"已收到"，Supabase contacts 表新增一条记录。
【证据材料】Vercel Logs 里出现 Missing SUPABASE_URL；截图见附件。
【不要改什么】不要改首页样式，不要重写整个表单，只修提交链路。
```

这比一句“表单坏了，帮我修”有效太多。上下文包不是给 AI 写作文，是让它少猜、少乱改、少返工。

## 有些问题你说不清楚：截图

有时候你知道哪里不对，但用文字描述很费劲。

“那个什么什么区块下面，有一个东西歪了，就是那个，你知道的那个。”

AI 不知道。

最简单的方法：截图。

我做 sevenailab 的时候养成了一个习惯。偶尔生成了 Bug，比如一个按钮点不动，不用慌，也不用读源码。直接截图丢给 AI：

“这里坏了，你自己修一下。”

很多时候，它能很快给出一版修复。

这是修改里最低成本的方式。你不需要知道这个按钮的代码写在哪个文件、用了什么 CSS 类名、绑了什么事件。你只需要告诉 AI：“看这张图，红圈标出来的地方有问题。”

如果你用 Cursor，还有一个更精准的方式：Stagewise 插件（一个浏览器扩展，装完后可以在浏览器里直接点击页面上的元素，Cursor 会自动定位到对应的代码位置）。哪里不爽点哪里，不需要你翻文件找代码。

但今天你可能还在用 Lovable。没关系，Lovable 本身就支持截图修改：把截图贴进聊天框，描述问题，它会识别并修复。

## 学一个必备技能：浏览器开发者工具

这里我要教你一个东西。很多零基础的人从来没打开过它，但它是“修”这个环节最有用的工具之一。

在浏览器里，按 F12（Windows）或 Cmd+Option+I（Mac）。

一个面板会弹出来。这就是“开发者工具”（DevTools）。

不要慌。你不需要看懂里面所有的东西。

你只需要知道两件事：

**第一，用它定位元素。**

点击开发者工具左上角的小箭头图标（或按 Cmd+Shift+C），然后用鼠标去点页面上你想改的元素，比如一个按钮、一段文字、一张图片。

你会看到这个元素在代码里对应的位置被高亮了。右边的 Styles 面板会显示它当前的样式：字号多大、颜色是什么、间距多少。

你甚至可以直接在 Styles 面板里改数字，实时看效果。改完觉得对了，再把你想要的数值告诉 AI：“把这个按钮的 padding 改成 16px，字号改成 18px，颜色改成 #2563EB。”

这比说“帮我把按钮改大一点”精确一万倍。

如果你看到某个样式有一条删除线划过去，说明它被另一条样式覆盖了。这是 CSS 最常见的“迷惑”：你明明设置了颜色，但不生效，因为有一条优先级更高的规则把它盖住了。遇到这种情况，截图给 AI，说“我设置了这个颜色但不生效，Styles 面板里显示有删除线，帮我看看是被什么覆盖了”。它能帮你解决。

**第二，用它看错误日志。**

切换到 Console（控制台）面板。如果页面有 JavaScript 报错，你会看到红色的错误信息。

你不需要看懂它。

你需要做的是：把这段红色文字复制下来，贴给 AI。

“我的页面出现了这个报错：[粘贴]。请帮我解释这是什么意思，并修复它。”

一个更好的问法，我自己经常用：

“就像给 5 岁孩子解释一样，告诉我这行报错是什么意思。请帮我解析在这个技术链路中每个部分负责的逻辑是什么。现在的错误发生在哪个环节，请给出具体的修复方案。”

这样问的好处是，你拿到修复方案的同时，也能顺带理解这个错误是怎么发生的。不是每次都要这么问。但当你发现某个报错反复出现，或者修了一个地方另一个地方又坏了的时候，这种“请解释链路”的问法会帮你建立系统感。

## CSS 是你最常改的那一层

上一章我们讲过三层理解：HTML 是骨架，CSS 是皮肤，JavaScript 是神经系统。

在“修”这个环节，你改得最多的是 CSS。

按钮太小、颜色不对、间距太窄、在手机上排版乱了，这些全是 CSS 的问题。

所以这里插一个最小的 CSS 调试知识。

开发者工具里的 Styles 面板，列出了当前元素所有生效的样式。你会看到这些东西：

```css
color: #333;           /* 文字颜色 */
font-size: 16px;       /* 字号 */
padding: 12px 24px;    /* 内间距 */
margin-bottom: 20px;   /* 下方外间距 */
background-color: #fff; /* 背景色 */
border-radius: 8px;    /* 圆角 */
```

不用记住所有 CSS 属性。在 Styles 面板里看到某个属性，双击它的值，直接改。改了马上就能看到效果。

## 别什么都交给 AI

说了这么多“让 AI 帮你改”，我要反过来说一句。

别过度依赖 AI。

我踩过这个坑。什么都丢给 AI，自己的脑子真的会变懒。

尤其是判断层面的事。“这个页面的布局合不合理？”“这个用户流程顺不顺？”“这个文案是不是我想传达的意思？”这些 AI 可以给建议，但最终判断必须是你的。

AI 可以帮你改代码、调样式、修 Bug。但它替不了你判断“这个东西改完之后，是不是真的更好了”。

所以每次 AI 改完，不要直接说“好”。用你的眼睛走一遍。自己判断。

这个习惯很重要：AI 说改完了，你要自己看一遍改完的结果，确认对了，才算完。

## 修改 prompt 三段式的进阶用法

当你用多了之后，你会发现三段式可以叠加使用。

一次性提交多个修改时，每条都用三段式：

```text
请修改以下问题：

1.【位置】首页 Hero 区块的主标题
  【现象】字号太大，在手机上两行变三行
  【期望】字号从 48px 降到 32px，手机上保持两行以内

2.【位置】导航栏
  【现象】在手机上横向溢出，出现滚动条
  【期望】手机上变成汉堡菜单

3.【位置】服务介绍的三张卡片
  【现象】鼠标悬停没有任何交互反馈
  【期望】悬停时卡片轻微上浮 + 出现阴影，参考 Stripe 官网的卡片效果

4.【位置】底部联系表单
  【现象】提交按钮点击后没有任何反馈
  【期望】点击后按钮显示 loading 状态，提交成功后弹出"已收到"的提示
```

你看到了吗？每一条都是独立的、无歧义的。AI 不需要猜任何东西。

这种清单式的修改 prompt，是我做几个项目后发现的效率最高的工作方式。一轮下来能修 5-10 个问题，比逐条聊天快得多。

## 五类报错分层

最后说一个心法级的工具。

遇到问题，第一反应往往是“坏了”。然后把一句“坏了”丢给 AI。

但“坏了”有很多种。按钮不见了和按钮在但点了没反应，是完全不同的问题。你能分清它属于哪一类，AI 就能更快地修。

我把产品里最常见的报错分成五类：




```text
【五类报错分层】

第一类：看得见但不对，CSS 层
  按钮在，但颜色不对 / 太大太小 / 间距不对 / 手机上布局乱了。
  → 告诉 AI 改样式。

第二类：看得见但不动，JavaScript 层
  按钮在，样子也对，但点了没反应 / 表单提交无效 / 页面不跳转。
  → 告诉 AI "这个按钮点了没反应，请检查点击后的逻辑"。

第三类：动了但数据没到，API / 数据库层
  操作有反馈，但刷新后数据消失 / 换设备看不到 / 表单提交了但后台没收到。
  → 告诉 AI 检查数据请求和数据库连接。

第四类：整个页面白了，构建 / 部署层
  页面打不开 / 全白 / 显示 404 或 500 错误。
  → 告诉 AI 检查构建日志和部署配置。

第五类：某些环境有问题，兼容性层
  电脑上没问题但手机上坏了 / Chrome 正常但 Safari 异常 / 本地正常但上线后出错。
  → 告诉 AI 你在什么环境下遇到问题，附上设备和浏览器信息。
```

不用精通每一层。能做到一件事就够：当某个东西不对的时候，说出“我觉得问题在第几类”。

比如“按钮在那里但点了没反应”，你就知道这不是第一类（CSS），而是第二类（JavaScript）。JavaScript 是页面的“神经系统”，负责处理用户操作后应该发生什么。最常见的原因是按钮和它背后的功能没有连接上。你不需要自己去修，只需要告诉 AI：“这个按钮点了没反应，应该是点击后触发 XX 功能，请检查并修复。”

再比如“我在手机上打开页面，某个区块的布局全乱了”，这是第五类。你可以告诉 AI：“这个问题只在手机 Safari 上出现，电脑上是正常的，请检查响应式和浏览器兼容性。”

这种“先分类再描述”的能力，就是系统观。不要只把报错丢给 AI。把上下文和完整日志喂给它，让它帮你定位是链路的哪一环出了问题。多数时候，理清楚分类，问题就解决了一半。

---

## 要点小结

修改的质量取决于你描述问题的质量。越精确，AI 改得越对，来回越少。不要逐个修 Bug，而是走完核心流程后列个清单，用“位置 + 现象 + 期望”的三段式一次性提交。遇到看不懂的问题就截图，打开开发者工具的 Styles 面板实时改数字。最关键的是分清问题在哪一层：CSS 层、JavaScript 层，还是数据层。这样告诉 AI 时，目标就清楚。

**本章行动**：走一遍你第一版的完整流程，把所有问题列成清单。用“修改 prompt 三段式”（位置 + 现象 + 期望）格式化，一次性提交给 AI 修改。
