# 第十七章 · Bug 不是灾难，是路线牌

> 所属部分：第四部分 · 打磨，让产品真的可用  
> 读完本章，你会得到：个人报错排查清单  
> 本章方法：最低成本首问法 + 报错三段读法

---

做伴聊的时候，我有一半时间在修 Bug。

先把这一章和第七章区分清楚：第七章讲“你知道哪里不对，要让 AI 改”，重点是怎么描述修改需求；这一章讲“你还不知道为什么不对，要先定位问题在哪一层”，重点是排查路径。

不是夸张。开发过程中至少一半时间在和各种错误打交道。功能写好了跑不通，跑通了发现边界情况没处理，处理了发现另一个地方又因为这次修改坏了。

刚开始我觉得这是灾难。“是不是我太笨了？”“是不是 AI 不行？”“正常人开发也这么多 Bug 吗？”

后来我才理解：这就是正常的。

专业程序员也会花大量时间处理 Bug。他们的优势不在于“永远不出错”，而在于知道怎么快速定位问题出在哪里。

这一章教你的就是这个：学会和 Bug 打交道。消灭不了，但可以快速定位、快速修。

---

## 最低成本首问法

遇到 Bug 的第一反应往往是慌。然后开始乱改，改这里看看、改那里试试、删了重来。

不要这样。

遇到 Bug 的第一步，是确认它属于哪一层。在第七章 我们已经学过“五类报错分层”。这一章我们把它变成一个更具体的操作流程。




```text
【最低成本首问法】

遇到 Bug 时，按顺序问自己这四个问题：

1. 是不是我看错了？
   → 刷新页面（Cmd+R）
   → 清除缓存（Hard Refresh：Cmd+Shift+R）
   → 换一个浏览器试试
   → 很多表象问题在这一步就能排除

2. 控制台说了什么？
   → 打开浏览器开发者工具（F12）→ Console 面板
   → 红色文字 = 错误，黄色文字 = 警告
   → 把红色错误文字复制出来

3. 这个错误属于哪一层？
   → CSS 层：看得见但样式不对（颜色、位置、大小）
   → JavaScript 层：能看到但点了没反应、数据不对
   → API/数据库层：数据加载失败、提交后没响应
   → 构建/部署层：本地能跑但线上打不开
   → 兼容性层：电脑上正常但手机上不正常

4. 我能把它给 AI 描述清楚吗？
   → 截图 + 错误信息 + “我期望的是 X，实际看到的是 Y”
   → 这三样东西给 AI，它就能帮你修
```

四个问题，从最简单的开始。每一步都不需要懂代码。

## 报错信息怎么读

控制台里的报错信息看起来很吓人，一大串英文、一堆文件路径、几十行堆栈信息。

但你只需要读三个部分：

```text
【报错三段读法】

第一段：错误类型（第一行）
  → TypeError: Cannot read properties of undefined
  → 这一行告诉你“出了什么类型的错”
  → 你不需要完全懂，只需要把这行复制给 AI

第二段：错误位置（at xxx.js:42:15）
  → 告诉你错误发生在哪个文件的哪一行
  → 你不需要去那一行看代码，但这个信息帮 AI 定位

第三段：错误上下文（前后几行）
  → 错误发生时，程序在做什么操作
  → 这个信息越多，AI 修得越准
```

实际操作：按 F12 → 看 Console 标签 → 找红色文字 → 复制第一行给 AI。你不需要完全理解报错信息，AI 会帮你翻译。

我做伴聊时遇到过一个错误：“Error: Cloud function execution failed”。我完全不知道这是什么意思。但我把这行错误加前后相关日志复制，发给 Cursor，Cursor 自动定位了问题并修改了代码。我只需要验证修复后错误是否消失。

这就是非技术人员修 Bug 的方法：把错误信息准确地传递给 AI，比看懂错误本身更重要。

---

## 像给 5 岁孩子解释

当你把错误信息发给 AI，它给你一段解释。

但有时候它的解释你也看不懂。

这时候用一个技巧：让 AI 把解释再简化一遍。

“请像给 5 岁孩子解释一样，告诉我这个错误是什么意思。”

AI 会把技术术语翻译成大白话。比如：

技术版：“TypeError: Cannot read properties of undefined (reading 'map')”。

5 岁版：“你让程序去一个叫 'map' 的地方拿东西，但那个地方根本不存在。就像你让小朋友去冰箱拿牛奶，但冰箱根本没有打开。程序需要先确认东西存在，再去拿。”

这个技巧对非技术人员非常有用。你不需要理解底层原理，只需要知道“问题大概是什么”，就能判断下一步该怎么做。

## Bug 在不同地方显示在哪

你的产品出了问题，错误信息不一定在同一个地方出现。取决于你用的是什么工具：




```text
【去哪找 Bug 日志】

在 Lovable 里开发：
  → 看 Lovable 编辑器右下角的错误提示
  → 如果是预览窗口里的问题，点 F12 打开 Console

在 Cursor 里开发：
  → 终端窗口里的红色文字（编译报错）
  → 浏览器里按 F12 → Console 面板（运行时报错）

产品已经上线（部署在 Vercel）：
  → 打开 Vercel 后台 → 你的项目 → Logs 面板
  → 这里能看到线上的报错日志

用手机打开产品发现有问题：
  → 用电脑浏览器打开同样的页面复现问题
  → 然后按 F12 看 Console
  → 手机上没法直接看日志，所以先在电脑上复现
```

不管在哪找到错误信息，操作都一样：把红色的错误文字复制下来，发给 AI（Lovable 的聊天框、Cursor 的 AI 面板、或者 Claude 对话），让它帮你修。你不需要看懂错误信息，只需要完整地复制给 AI。

## 常见 Bug 的解决策略

把我做五个项目遇到的常见 Bug 分个类，给你一份速查：

**页面修改不显示**：最常见的原因是文件没保存（Ctrl+S / Cmd+S）或浏览器没刷新（Cmd+R）。超过一半的“Bug”是这个原因。

**按钮点了没反应**：告诉 AI“这个按钮点击后应该执行 X，但点了没反应”，AI 会帮你检查按钮和功能之间的连接有没有接上。

**数据加载失败**：页面空白或者显示“加载失败”。打开 F12 → Console，复制红色错误信息给 AI。

**样式不对**：颜色、大小、位置和你预期的不一样。截图发给 AI，说“我期望的是 X，实际看到的是 Y”。

**线上打不开但本地正常**：去 Vercel 后台看 Logs，把报错复制给 AI。最常见的原因是环境变量没配。

**编译报错**：终端里一片红。把完整的报错文字复制给 AI，让它帮你修。不用自己读。

这些你不需要记住。遇到了对照查就行。

## 不要过度依赖 AI 的判断

最后说一个容易忽略的事。

AI 修 Bug 往往很快。你把错误信息给它，它通常能给出修复建议。

但有时候 AI 修的只是“错误的表象”，不是“真正的问题”。

比如一个页面加载很慢。AI 可能帮你压缩了图片、减少了 CSS。但真正的问题是一个 API 调用在循环里重复了 100 次。AI 修的是症状，不是病因。

所以你需要保持一个习惯：AI 修完了，你验证。除了看“错误消失了没”，还要看“整个功能正不正常”。

还有一个更深层的提醒：不要让 AI 帮你做所有判断。在一些关键问题上，比如“这个功能要不要砍”“这个 Bug 影不影响核心流程”“这个修复方案会不会破坏其他地方”，你必须自己想。

---

## 要点小结

Bug 是正常的，专业程序员也会花大量时间处理 Bug。关键是快速定位问题属于哪一层，把准确的信息传递给 AI。学会“报错三段读法”，你就能准确复制错误信息，这比完全看懂错误更重要。

**本章行动**：打开你的产品故意触发一个错误（表单不填就提交、断网刷新）。打开 F12 → Console 找到红色文字，用报错三段读法拆解它，复制给 AI 让它修复。
