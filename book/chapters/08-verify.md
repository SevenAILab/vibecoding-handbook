# 第八章 · 验：看不出对错，AI 就在帮倒忙

> 所属部分：第二部分 · 主控流程，定生修验发察  
> 读完本章，你会得到：上线前检查清单  
> 本章方法：验收五步法 + 设备检查矩阵

---

我做 Say it 的时候犯过一个错。

AI 帮我写了一个很完整的用户设置页，包含配色选择、语言切换、通知偏好、深色模式。精美极了。我看着觉得挺好的，就接着做下一步了。

直到我把链接发给一个朋友。朋友打开后问了一句：

“这个东西是干什么的？”

那一刻我意识到，我连核心流程都没有验收过。整个页面看起来很完整，但最重要的功能（用 AI 采访你然后生成内容）根本走不通。

从那以后我学会了一件事：AI 做完了不等于做对了。你必须自己验收。

---

## AI 可以当程序员，但你必须当产品经理

AI 写代码确实快。但它也能非常快地把一个错误方向写得特别完整。

它不会停下来问你“这个真的需要做吗？”。它只会执行。你说加什么，它就加什么。你不说停，它就不停。

所以验收这件事，AI 替代不了你。

你是产品经理。你来判断什么算对、什么不算对、什么该留、什么该砍。

---

## 验收五步法

我把验收拆成五步。每次改完一轮，走一遍这五步：




```text
【验收五步法】

1. 走一遍核心流程
   从"打开页面"到"完成核心动作"，一步不跳地走完。
   → 能走通吗？哪一步卡了？

2. 换一个设备
   电脑上做的？用手机打开看看。
   → 排版正常吗？按钮能点吗？文字会不会溢出？

3. 假装自己是新用户
   清掉你脑子里的预设。假装你是第一次打开这个页面。
   → 3 秒内能看懂这是干什么的吗？知道下一步该做什么吗？

4. 测试失败状态
   表单什么都不填就提交，会怎样？网络断了会怎样？
   → 用户看到的是友好提示，还是一个白屏？

5. 问一个真人
   把链接发给一个人，什么都不解释。看他们怎么用。
   → 他们卡在哪里了？他们的第一反应是什么？
```

这五步不用按测试工程师的标准来做。它们是产品人的最小验收动作。

你不需要做压力测试、不需要写单元测试、不需要检查所有边界条件。但这五步你必须做，因为它们覆盖了“真人用你的产品时最可能遇到的问题”。

---

## 手机上看，是最容易被忽略的一步

你在电脑上做产品。你盯着一个宽屏看效果。一切都很好。

然后你把链接发给朋友。朋友用手机打开。

标题挤成两行了。按钮缩到看不见。图片超出屏幕。导航栏叠在一起。

这个问题叫“响应式适配”。

响应式是什么意思？就是页面能根据屏幕大小自动调整布局。大屏上三列排列的卡片，到手机上可能变成一列纵向排列。

好消息是，如果你用 Lovable，很多响应式是默认就有的。坏消息是，“默认有”不等于“一定对”。

所以验收时一定要做一件事：用手机打开你的产品。

比如你在手机上看到布局乱了。你不需要知道代码怎么写，只需要告诉 AI：“在手机上这个区块应该是单列排列，在电脑上是三列。”AI 会处理具体实现。

如果你暂时没有真手机测试环境，可以用浏览器的模拟功能：打开开发者工具（F12），点击左上角的“设备切换”图标（一个手机和平板的小图标），浏览器会模拟手机屏幕宽度。

---

## 失败状态是产品的真面目

第四步“测试失败状态”，很多人跳过了。

因为谁会故意去把自己的产品弄坏呢？

但你的用户会。

不是故意的。他们会在表单里什么都不填就点提交。会在网速很慢的时候反复点按钮。会输入你没预期的内容：一段特别长的文字、一个特殊符号、一个空格。

你没法预防所有失败。但你能确保一件事：当失败发生时，用户看到的是一个友好的提示（“请填写必填项”“提交失败，请稍后重试”“加载中，请稍候”），而不是白屏或一段看不懂的英文报错。

怎么做？最简单的方式：你自己试一遍失败情况，然后告诉 AI：

“当表单为空提交时，需要提示'请填写必填项'。”

“当网络请求失败时，需要显示一个重试按钮，而不是白屏。”

“按钮点击后应该立即变成 loading 状态，防止用户重复点击。”

还有一个容易忽略的点：用户出错之后，能不能自然地重试？按钮是不是卡在 loading 状态不回来了？页面是不是需要手动刷新才能继续？失败提示不只是告诉用户“出错了”，还要告诉他“你可以这样继续”。

---

## 让 AI 帮你检查

验收不只是你一个人的事。AI 也可以帮忙。

我做 sevenailab 的时候发现一个特别好用的功能：Lovable 在发布前会自动扫描网站，给出“安全”和“速度”报告。比如“图片太大，影响加载速度”或“这个 API 有安全隐患”。

最关键的是，你不需要懂这些报告在说什么。只要点“尝试修复（Try to fix）”，它会自己改。

这让“上线前审一遍”这件原来高门槛的事，变成了每个人都能做的一步。

你还可以主动让 AI 帮你验收。上一章提过反向提问的方法，这里同样适用：

“你觉得这个页面还有什么可以优化的地方？请从用户体验、性能、移动端适配三个角度分别检查。”

AI 会给你一份清单。你不需要全部采纳，但它经常能发现你漏掉的东西。

如果后面你用到 Claude Code 这类更强的 AI 工具，它有一个很好的设计：修改文件前总是先问你批准。你可以逐个审批改动，也可以开启“Accept all”模式让它自己跑。但即使自动化程度再高，关键决策权始终在你手里。这个原则在验收环节尤其重要：AI 可以跑检查，但“通不通过”这个判断，你来下。

## 真实部署的验收

如果你只在 Lovable 的预览窗口里看效果，那只算看了一半。

真实部署一定要尽早。不要永远停留在本地预览里自嗨。

很多问题在本地永远发现不了。只有在真实环境里它们才会跑出来：数据库连接会不会断？登录流程会不会出问题？页面加载速度够不够快？不同浏览器里的表现是不是一样？

这个过程确实烦。但绕不过去。一个产品从“能跑”到“能用”，中间隔着一堆很现实的问题。

好在 Lovable 的发布流程很简单，点发布按钮就行。拿到 URL 后，用手机打开看一遍，就是最真实的验收。

---

## 设备检查矩阵

我给你一个简单的检查矩阵。每次发布前，至少测试这些组合：




```text
【设备检查矩阵】

         电脑浏览器  手机浏览器  
核心流程      ✓          ✓
首屏可读性    ✓          ✓
按钮可点击    ✓          ✓
表单可提交    ✓          ✓
失败状态      ✓          ✓
加载速度      ✓          ✓
```

不需要测试所有浏览器、所有手机型号。但至少保证：“电脑上能用”+“手机上能用”。

如果你有条件，让一个用 iPhone 的人和一个用安卓的人各试一遍。不同手机浏览器的表现偶尔会有差异。

没有真手机测试？在浏览器里按 F12 打开开发者工具，点左上角的手机图标（或按 Cmd+Shift+M），就能模拟手机屏幕。

## 验收心态

最后说一个心态。

验收不是“挑毛病”。它是在确认：这个东西是不是真的能给人用。

做完一版后，你心里可能有点虚。不太敢给别人看。怕被说“这也太简陋了”或者“这有什么用”。

第一版的标准就两条：核心流程走得通，别人看得懂这是干什么的。够了。

只要你走完验收五步法，能回答以下三个问题，这一版就过关了：

```text
1. 核心流程能不能从头走到尾？
2. 手机上能不能正常用？
3. 有没有至少 1 个人试过并告诉你他们的感受？
```

能回答“是”，就可以准备发布了。

---

## 要点小结

验收是确认产品能真的给人用。AI 可以执行任何指令，但做得对不对要你来判。用五步法走一遍：核心流程、换设备、新用户视角、失败状态、真人反馈，每一步都有可能发现问题。过关标准就三条：核心流程能走通，手机上也能用，有真人看过。

**本章行动**：用验收五步法走一遍你的产品，用手机打开检查布局，把链接发给 1 个人看看他们的反应。
