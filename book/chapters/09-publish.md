# 第九章 · 发：不上线就没有真实世界

> 所属部分：第二部分 · 主控流程，定生修验发察  
> 读完本章，你会得到：一个公网可访问的产品 URL  
> 本章方法：部署四层图 + 域名配置清单

---

我做 7brandhub 的时候，有一段“鬼打墙”的经历。

产品在本地跑得很好。代码也推到了 GitHub。Vercel 也部署了。域名也买了。

但打开域名时白屏。什么都没有。

我以为代码坏了，反复检查。AI 帮我查了一圈，代码没问题。后来发现原因荒唐得很：域名的 DNS 设置里，一个 IP 地址填错了一位数。

这个经历让我意识到一件事：从“本地能跑”到“公网能打开”，中间隔着好几层东西。你不需要精通每一层，但你要知道它们存在。否则出了问题，你连查哪里都不知道。

这一章，我们把产品推到公网上去。

---

## 如果你用 Lovable：三步发布

Lovable 内置了发布功能，操作很简单：

```text
第 1 步：打开你的 Lovable 项目，点击右上角的“发布（Publish）”按钮。
第 2 步：等待页面生成 URL，格式类似 xxx.lovable.app。
第 3 步：复制这个 URL，用手机浏览器打开，确认页面正常。
```

到这一步，你的产品已经在公网上了。任何人输入这个链接都能看到。

如果你想用自己的域名（比如 yourproduct.com），需要多做几步，后文会讲。

如果你一直在用 Lovable 做产品，可以先跳到后面的“域名”部分。但如果你开始用 Cursor，或者想理解“发布”这件事到底经历了什么，接下来这部分很重要。

---

## 部署四层图

“发布”听起来简单，但背后有四层东西在工作。你不需要精通每一层，但理解每一层的作用会帮你快速排查问题。




```text
【部署四层图】

第一层：代码
  你的产品代码，存在你的电脑里（如果用 Cursor）
  或存在 Lovable 的云端（如果用 Lovable）。

第二层：代码仓库（GitHub）
  代码的“云备份”。
  每次改完代码，你把它推送到 GitHub 上。
  这样你有了历史记录、版本管理、多人协作的能力。

第三层：部署平台（Vercel）
  连接你的 GitHub，自动把代码变成一个可访问的网站。
  每次你推送新代码到 GitHub，Vercel 自动重新部署。
  这里还会发生一次自动构建（build）：把源代码变成浏览器能理解的网页。

第四层：域名（Namecheap 等）
  给你的网站一个人类能记住的地址。
  把 yourproduct.com 指向 Vercel 的服务器。
```

Lovable 帮你把这四层全包了。你点发布，它帮你搞定代码存储、构建、部署。

但如果你用 Cursor，你需要自己走一遍这四层。

如果你一直在用 Lovable，这四层已经被自动处理了，你只需要关注第四层（域名）。如果你后面用 Cursor 开发，才需要自己处理第二、第三层。

---

## 第一层：代码（你已经有了）

如果你用 Cursor 开发，代码就在你电脑的一个文件夹里。

确保它能在本地跑起来：打开 Cursor 的终端，输入 `npm run dev`，浏览器打开 localhost:3000，页面正常显示。

然后做一件事：在发布前，先运行 `npm run build`。

这一步是“编译检查”。它会告诉你代码有没有致命错误。如果运行成功，说明代码质量过关，可以上传。如果报错了，把报错信息复制给 AI，让它帮你修。

这是一个好习惯：先在本地编译通过，再推到线上。

---

## 第二层：GitHub

GitHub 是什么？简单说，它是代码的云端仓库。

你可以把它理解成“代码版的 iCloud”或“代码版的 Google Drive”。你的代码存一份在本地，同时同步一份到 GitHub 上。

为什么需要它？三个原因：备份（电脑坏了代码不丢）、历史记录（随时回退到之前的版本）、连接部署平台（Vercel 需要从 GitHub 拉代码）。

这里有几个概念你会反复听到：

仓库（Repository）像一个代码相册，记录你所有的修改。提交（Commit）就是保存一个版本并写上说明，比如“修复了首页按钮的颜色”“加了联系表单”。推送（Push）是把本地的修改上传到 GitHub。拉取（Pull）是把 GitHub 上最新的代码下载到本地。

最简单的用法是装一个 GitHub Desktop，一个图形界面的工具。操作流程：下载安装 → 用 GitHub 账号登录 → 把你的项目文件夹拖进去 → 修改文件后它会自动显示变化 → 点“提交”写一句说明 → 点“发布”上传到 GitHub。

全程不需要敲命令。

---

## 第三层：Vercel

Vercel 是一个部署平台。“部署”就是把你电脑上做的网页上传到服务器，让全世界任何人都能打开。它的核心能力是：连接你的 GitHub，自动把代码变成一个可以访问的网站。

具体操作步骤：

```text
第 1 步：打开 vercel.com，点右上角“Sign Up”。
       选“Continue with GitHub”，用你的 GitHub 账号登录。

第 2 步：登录后，点“Add New...” → “Project”。
       页面会列出你 GitHub 上的所有仓库。
       找到你的项目，点旁边的“Import”按钮。

第 3 步：Vercel 会问你一些配置选项。
       大多数情况下不用改任何东西，直接点“Deploy”。
       等 1-3 分钟，看到“Congratulations”就成功了。

第 4 步：复制 Vercel 给你的 URL（格式是 yourproject.vercel.app）。
       用手机打开，确认页面正常显示。
```

之后每次你把新代码推送到 GitHub，Vercel 会自动重新部署。你不需要再手动操作，改了代码，推上去，网站自动更新。

Vercel 还有几个对新手友好的能力：每次推送都会生成一个预览链接，你可以在正式上线前先看看效果；出了问题可以秒级回滚到上一个版本。

这里多说一句“构建”。Vercel 收到你的代码后，不是直接把代码原样丢给用户，而是先自动构建（build）一次。你可以把它理解成“打包出厂”：把源代码、样式、配置整理成浏览器能稳定访问的网页。构建失败，网站就不会更新。所以本地先跑 `npm run build`，就是提前模拟这一步。

## 环境变量：把秘密留在后台

这里要讲一个你马上就会遇到的概念：环境变量。

什么是环境变量？就是“不能写在代码里的秘密信息”。

比如你的 AI 模型 secret key、数据库高权限连接信息、支付密钥。这些东西如果直接写在代码里，推到 GitHub 上就相当于公开给全世界看了。有些前端可用的 public key / publishable key 本来就可以暴露，但也必须配好权限策略，不能把它当成安全边界。

所以你需要把它们存在一个叫 `.env.local` 的文件里（本地开发时）或者存在 Vercel 的环境变量配置里（上线后）。

在 Vercel 后台，找到你的项目 → Settings → Environment Variables，把 secret key、支付密钥等服务端配置填进去。Vercel 会在构建或运行时把这些变量提供给你的应用，但它们不会出现在公开的代码仓库中。

一个关键提醒：`.env.local` 和其他 `.env*` 文件不要上传到 GitHub。大多数项目模板里已经在 `.gitignore` 文件中排除了它，但你最好确认一下。

如果你用 Lovable，环境变量可以在 Lovable 的设置面板里配置，不需要手动创建文件。

环境变量的完整配置步骤和常见问题排查，会在第十六章详细展开。这里你只需要知道它是什么、为什么重要，以及它不能直接写进公开代码里。

## 第四层：域名

默认你的网站地址是 `yourproject.vercel.app` 或 `xxx.lovable.app`。

这能用，但如果你想让产品看起来正式一点，你需要一个自己的域名。

买域名的流程很简单。我用的是 Namecheap，海外比较主流。国内的话阿里云（万网）和腾讯云也可以。搜索你想要的域名，看看有没有被注册，选一个合适的，结账。.com 的域名一年大概几十块到一百多。

买完之后，你需要做一件事：把域名指向你的部署平台。

这一步叫 DNS 配置。DNS 你可以理解成互联网的电话簿，它告诉浏览器：“当用户输入 yourproduct.com 时，应该去找哪台服务器。”

我做 sevenailab 的时候，在 Namecheap 买好域名后，Lovable 提供了“自动连接”选项。授权访问 Namecheap 账户后，它自动配置 DNS 的 A Record 和 CNAME，再把域名设为主域名。

非技术人员最头疼的“最后一公里”，在这种托管流程里会被大幅简化。

但如果你用 Vercel + 手动配置，流程是这样的：




```text
【域名配置清单】

1. 在 Vercel 后台 → Settings → Domains → 添加你的域名
2. Vercel 会告诉你需要配置哪些 DNS 记录
3. 去域名商后台（Namecheap/阿里云等）→ DNS 设置
4. 添加 A Record 和 CNAME Record，填入 Vercel 给的值
5. 等待 DNS 生效（通常几分钟到几小时）
6. 回到 Vercel 确认域名已连接
```

我做 7brandhub 时踩过一个坑：域名的 DNS 没有正确指向 Vercel 的官方 IP。导致域名解析失败，页面打不开。

这个问题对程序员来说可能只是改一行配置。但对我来说当时就是“鬼打墙”：明明域名买了，明明 Vercel 也设了，就是打不开。

最后怎么解决的？向 AI 描述清楚完整的链路：域名在 Namecheap，部署在 Vercel，DNS 设置了但不生效。AI 帮我一步步排查，最终发现是 DNS 记录填错了一个值。

这就是为什么你需要理解“部署四层图”。理解了四层结构，出问题时你至少能说出“我觉得问题在 DNS 这一层”，而不是对着屏幕干瞪眼。

### DNS 配置常见问题

DNS 配置最折磨人的地方，是它不一定立刻生效。多数情况下几分钟到几小时就能看到结果，少数情况可能需要 24-48 小时。你刚改完就刷新打不开，不一定是你配错了，可能只是还在生效中。

怎么判断？先回到 Vercel 的 Domains 页面，看它提示的是“缺少记录”“记录值不匹配”，还是“等待生效”。如果提示记录值不匹配，就去域名商后台核对 A Record 和 CNAME，一个字符都别差。如果 Vercel 显示没问题，但你电脑打不开，可以换手机网络试一次，或者把域名发给朋友打开，排除本地缓存问题。

最常见的错误有三个：第一，A Record 填错了 IP；第二，根域名和 `www` 只配了一个，另一个没配；第三，之前配过旧平台的 DNS 记录，没有删干净。遇到这种情况，把 Vercel 的域名提示页和域名商的 DNS 页面截图一起发给 AI，比你自己猜快得多。

## 我做 sevenailab 时的完整部署路径

给你看一个真实的完整路径，作为参考：

1. 在 Namecheap 买了 sevenailab.com。
2. 在 Lovable 里开发完产品。
3. Lovable 点发布，拿到 .lovable.app 的临时链接。
4. 在 Lovable 的域名设置里，连接 Namecheap，自动配置 DNS。
5. 等几分钟，sevenailab.com 就能访问了。

如果你用 Cursor + Vercel 的路径，则是：

1. 在 Namecheap/阿里云买域名。
2. 在 Cursor 里开发，`npm run build` 检查无误。
3. 用 GitHub Desktop 把代码推到 GitHub。
4. 在 Vercel 里导入 GitHub 项目，自动部署。
5. 在 Vercel 添加域名，去域名商配置 DNS。
6. 等几分钟，域名生效。

两条路径的终点一样：一个公网可访问的 URL。

## 如果卡住了

部署这一步最容易卡在两个地方。

第一个是环境变量。你明明在本地跑得好好的，一部署就报错。九成是因为环境变量只存在本地的 `.env.local` 文件里，没有在 Vercel 或 Lovable 的后台配置。解决办法：打开部署平台的设置页面，把 `.env.local` 里的每一行都手动填进去。

第二个是 DNS 配置。域名买了，Vercel 也设了，就是打不开。不要自己猜，直接把以下信息一起发给 AI：“我的域名在 XX 平台买的，部署在 XX 平台，DNS 我配了这些记录（截图），但访问时显示 XX 错误。”信息给全了，AI 基本能一次帮你定位到问题。

## 发布不是终点

发布之后，你会有一种奇妙的感觉：这个东西真的在了。

任何人输入这个网址，都能看到你做的东西。

但发布不是终点。它是起点。

上线后的前 48 小时，不要立刻埋头加功能。先做四件小事：自己用手机和电脑各走一遍完整流程；发给 3 个目标用户或朋友，让他们边用边说感受；确认表单、登录、支付或核心按钮都能正常触发；记录第一个真实用户卡住的位置。这个阶段的目标不是把产品做大，而是确认它没有在门口把人挡住。

从这一刻起，你进入了真实用户反馈的阶段。真实用户会用你没想过的方式打开它。会在你没预期的设备上看它。会做你没测试过的操作。

这些都是好事。因为只有发布后，你才能收到真实反馈。

## 要点小结

发布是从“给自己看的练习”到“给真人用的产品”的分界线。理解部署四层图的目的很简单：出问题时知道查哪里。如果用 Lovable，前三层通常会被平台简化，你主要处理域名。如果用 Cursor，需要自己推代码上 GitHub，连接 Vercel 部署，配置域名 DNS。

**本章行动**：如果用 Lovable，发布产品拿到 URL；如果用 Cursor，走完 GitHub → Vercel → 域名 DNS 的三步。用手机打开公网 URL 验证一切正常。
