# 第六章 · 生：第一版要快，不要完美

> 所属部分：第二部分 · 主控流程，定生修验发察  
> 读完本章，你会得到：一个可交互的产品原型  
> 本章方法：最小闭环检查表

---

需求文档写好了，接下来只做一件事：把它变成一个能点、能交互、能看的东西。

要的是一个真的在浏览器里跑起来的页面，不是 PPT、不是效果图。

而且，这一版不需要完美。

---

## 最小闭环

上一章你已经用 PRD 砍过一遍了。这一章换个角度：拿着你的 PRD，用最小闭环清单再检查一次。

为什么？第一版只做一件事：验证这个东西有没有人愿意用。做完整产品是第二版的事。

核心闭环跑不通，再多功能也没意义。

MVP 是 Minimum Viable Product，中文叫最小可行产品。它不是“半成品”，而是一个能让早期用户体验、使用并反馈意见的版本。你用最小投入拿到用户反馈，判断这个产品到底能不能解决问题。

所以在让 AI 动手之前，先用这个清单检查一下你的需求文档：

```text
【最小闭环检查表】

1. 用户从打开到完成，最少几步？（目标：≤5 步）
2. 核心功能是不是只有 1 个？（不是 3 个，是 1 个）
3. 有没有可以砍掉的功能？（登录？设置？深色模式？多语言？）
4. 用户第一次打开，能不能不注册就体验？
5. 这个范围能不能一次性让 AI 生成？（太大就继续砍）
```

如果你的需求文档里有超过 3 个核心功能，砍到 1 个。




如果用户必须注册才能试，把注册去掉。

如果有“以后可能有用”的功能，全部删掉。

别一个 Bug 一个 Bug 修。列个清单让 AI 一次性改，效率高得多。这个技巧留给后面“修”那一章。现在先让第一版跑起来。

---

## 把需求文档喂给 AI

打开 Lovable（或 Bolt.new，或 Cursor，用你上手的那个）。

把上一章写好的需求文档贴进去。

如果你用 Lovable，确保在 Agent mode（不是 Plan mode），把需求文档贴进去，点发送。

通常几十秒到几分钟内，你会在右边的预览窗口看到一个完整的页面。

这一刻可能会让你兴奋。

先走一遍用户流程，用你的眼睛看看。

打开页面 → 核心功能能不能用 → 最终结果能不能看到。

如果核心流程能走通，这一版就及格了。

颜色、文字、按钮位置这些细节，后面都能改。核心流程通了就赢了。

## 第一次生成，常见的几个坑

听起来很顺对吧？贴进去、点发送，很快看到页面。实际操作时，你大概率会撞上这几个问题：

**需求文档太长，AI 只做了前半段。** 如果你的 PRD 超过一屏，AI 有可能只处理了开头的部分，后面的功能直接没做。解决办法：把 PRD 拆成两到三段分别发，或者在开头加一句“以下是完整需求，请全部实现”。

**第一版出来了但核心功能是假的。** 页面看起来很完整，有按钮、有表单、有弹窗，但点了以后什么都不发生。这是因为 AI 先做了 UI（骨架和皮肤），还没接逻辑（神经系统）。这时候不要急着改颜色改布局，先跟 AI 说“XX 按钮点了之后应该发生什么”，把核心交互接通。

**AI 自作主张加了你没要的东西。** 你写的是一个预约页面，AI 给你加了用户登录、深色模式切换，甚至一个博客板块。这就是为什么 第五章的 PRD 里“不做什么”那一栏很重要。遇到这种情况，直接告诉 AI：“删掉登录功能和博客板块，这一版不需要。”

**生成失败或者报错了。** 偶尔会遇到。最常见的原因是需求描述里有歧义，AI 不知道该怎么处理。比如“做一个像小红书一样的内容平台”，这句话对 AI 来说太大了，它不知道从哪下手。把范围缩小到“做一个可以发布图文帖子的页面，每个帖子有标题、图片和正文”，成功率会高很多。

这些问题都正常。第一次生成不完美，不代表工具不行，也不代表你的需求写得不好。改两三轮之后你就会摸到手感。

## 你刚看到的页面是怎么回事

这里我要插一段“最小技术理解”。

你不需要成为程序员。但你需要知道 AI 刚才生成的那个页面，是由什么组成的。

想象一个人的身体：

**HTML 是骨架。** 它决定页面上有什么：标题、段落、图片、按钮、表单、列表。每个元素是一个“标签”（tag），比如 `<h1>` 是大标题，`<p>` 是段落，`<button>` 是按钮。如果你在 Lovable 的代码面板里看到这些，不要害怕，它们就是积木块。

**CSS 是皮肤。** 它决定页面长什么样：颜色、大小、间距、字体、阴影、圆角。HTML 说“这里有一个按钮”，CSS 说“这个按钮是蓝色的、圆角的、点击时会变深”。

**JavaScript 是神经系统。** 它决定页面怎么动：点按钮发生什么、表单提交后跳转到哪、数据怎么传递。

你不用会写它们。但当你在 Lovable 的代码面板里瞥一眼时，你可以感受一下三层长什么样。以一个“预约”按钮为例：

```html
<!-- HTML：骨架，页面上有一个按钮 -->
<button class="btn-primary">预约诊断</button>
```

```css
/* CSS：皮肤，这个按钮长什么样 */
.btn-primary {
  background-color: #2563EB;
  color: white;
  padding: 12px 24px;
  border-radius: 8px;
}
```

```javascript
// JavaScript：神经系统，点击后发生什么
button.addEventListener('click', () => {
  openBookingForm();
});
```

不用看懂每一行。记住这个对应关系：改颜色/大小/间距 = CSS；改点击后发生什么 = JavaScript；加新元素 = HTML。




这个理解能帮你更快描述问题。比如“按钮在那里但点了没反应”，你就知道大概率不是 HTML 或 CSS 的问题，而是 JavaScript 的问题（交互逻辑没接上）。AI 就能更快定位。

## 第一版的标准

做完后，用这个标准衡量你的第一版：

```text
✓ 核心流程能走通（用户从头到尾能完成一个动作）
✓ 首屏能看出这是干什么的
✓ 功能不超过 3 个
✗ 不需要：完美的颜色/字体/间距
✗ 不需要：注册/登录
✗ 不需要：支付
✗ 不需要：数据持久化（第一版数据丢了没关系）
```

这一版的任务是“能用”。

我做 sevenailab 的时候，两天跟 AI 聊了 100 多条消息。起点很简单：先把这个业务的官网立起来，能给客户看、能收预约。小的、具体的、能上线的那一版，就是那个阶段要的全部。

你的第一版也是一样。

不过，快不等于裸奔。我现在让 AI 生成第一版时，会顺手加一句：帮我把加载中、出错、没有数据、操作成功这四种状态都处理了。




没有 Loading，用户点了按钮以为卡死了，连续点五次，后台可能跑了五遍。没有 Error 提示，用户以为是自己操作错了，反复试还是不行，然后关掉走人。没有 Empty 状态，页面上光秃秃的，用户不知道是还没加载还是真的没内容。没有 Success 反馈，用户提交了表单，页面纹丝不动，他不知道到底成功了没有。

你不需要把这些做得精致。一个转圈动画、一句“出了点问题，请稍后再试”、一行“还没有内容，去创建第一条吧”、一个绿色的“提交成功”就够了。但它们必须在那里。这不是在追求完美，是在让“能用”这件事真的成立。

## 要点小结

第一版只做一件事：验证一个最小闭环能不能跑。砍掉登录、砍掉设置、砍掉所有“以后可能有用”的东西。用户从打开到完成，最少要走几步，那就是你的第一版范围。核心功能只有 1 个，路径只有 1 条。

**本章行动**：用最小闭环检查表（5 个问题）检查你的需求文档，确保核心功能只有 1 个。把文档贴进 Lovable，生成第一版。用眼睛走一遍用户流程：能从打开走到完成吗？
