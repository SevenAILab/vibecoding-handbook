# 第二十一章 · 实战 4：行业平台与 Agent Demo

> 所属部分：第五部分 · 实战，四章五个真实案例  
> 本章路线：7brandhub = 国内行业；ClawChat = 全球 Agent 前沿  
> 本章方法：行业平台核心三问 + 探索型 Demo 诚实检查

---

最后两个案例放在一起讲，因为它们代表了两个完全不同的方向。

7brandhub 是“重”的。行业平台，需要数据库、需要导入几百条公司信息、需要搜索和分类、需要深度行业经验。

ClawChat 是“轻”的。一个探索性 Demo，龙虾和龙虾聊天，想法很怪，但它触碰了 Agent 社交的前沿。

两个项目教给我的东西完全不同。

所以这一章分上下半场读。上半场看 7brandhub，重点是行业平台：数据、分类、行业理解。下半场看 ClawChat，重点是探索型 Demo：前沿假设、投入边界、需求验证。你不需要把这两个方向混在一起，它们只是刚好提醒你：产品可以很重，也可以很轻，但都要诚实面对自己的验证目标。

```text
【案例复盘卡：7brandhub + ClawChat】

起点：一个来自行业经验，一个来自前沿想象
关键动作：平台先问行业三问，Demo 先问诚实检查
最大坑：平台容易低估数据质量，Demo 容易把“有趣”误判成“有需求”
关键判断：行业平台拼认知深度，探索型 Demo 拼假设验证速度
可复用动作：先判断你的想法属于“行业型”还是“探索型”，再决定投入上限
```

---

## 上半场：7brandhub

### 为什么做

我做品牌策略出身。一直觉得品牌行业有一个痛点：品牌方找服务商很难。不知道有哪些公司、谁做什么、水平怎么样。信息散落在各处，没有一个聚合的地方。

7brandhub 就是一个品牌服务商聚合平台，把几百家品牌服务公司的信息整理在一起，可以按类型、行业、地区搜索。

### 行业平台核心三问

做行业平台之前，你需要回答三个问题：

```text
【行业平台核心三问】

1. 你对这个行业有多深的理解？
   → 行业平台不是技术问题，是行业认知问题
   → 数据怎么分类、什么信息最重要、用户搜什么，这些 AI 替你想不了

2. 数据从哪里来？
   → 你有现成的数据（比如我有 120 家公司的 Excel）？
   → 还是需要从零收集？
   → 数据的质量和数量直接决定平台有没有价值

3. 为什么用户要来你这里而不是 Google？
   → 你的聚合提供了什么 Google 搜不到的价值？
   → 筛选条件？行业评价？对比功能？
```

如果这三个问题你答不上来，先别急着做平台。

我见过不少人想做“XX 行业的聚合平台”，但答不上第三个问题：用户为什么不直接 Google？如果你的回答是“因为我把信息整理在一起了”，这还不够。Google 也能搜到。你需要的是 Google 搜不到的维度，比如行业内部的口碑评价、真实案例对比、定制化的筛选条件。

我能答上来是因为我在品牌行业做了很多年。我知道怎么分类、什么信息对品牌方有用、什么搜索维度最关键。这些不是 AI 能帮你凭空想出来的。

### 技术复杂度

7brandhub 的技术复杂度比 sevenailab 和伴聊都高。

它需要：数据库存储几百条公司信息、搜索和筛选功能、分类标签系统、每个公司的详情页、数据批量导入。

我用了多个工具协作：用 Manus 参考网站风格快速生成 UI、用 Codex 做核心开发、用 Supabase 做数据库。

最大的坑是数据库格式兼容性。在对接数据库时遇到两个问题：一是数据库格式不兼容，工具建议换 Turso 数据库，但后来在 Vercel 里错误地链接了两个数据库导致冲突。二是域名 DNS 解析没有指向正确的 IP。

解决方案很简单：向 AI 提问完整的逻辑链条。“从域名到 Vercel 到 GitHub 到数据库，这几层到底怎么串联的？哪一环断开了？”AI 帮我画出了完整链路，我一眼看到了断点。

### 行业平台的教训

7brandhub 教给我的最重要的事：行业平台需要的是行业深度，技术能力反而是次要的。

AI 可以帮你写代码、搭数据库、做搜索功能。但“这个行业的公司应该怎么分类”“用户最关心哪些信息”“搜索结果按什么排序”，这些决策需要你对行业有真实的理解。

如果你在某个行业有 3 年以上的经验，行业平台是一个很好的产品方向。如果你没有，先做更小的东西：官网、工具、轻 SaaS。

## 下半场：ClawChat

### 龙虾的诞生

ClawChat 的灵感不是分析出来的。

它是在半睡半醒的时候突然冒出来的一个画面：一个很像微信的聊天界面，但里面聊天的是我的龙虾和别人的龙虾。

那种感觉有点像脑子里突然闪过一个自己都觉得很怪、但又莫名很想马上试出来的东西。

这就是 ClawChat 的起点。不是市场分析，不是用户调研。就是一个画面。

### Agent 社交是什么

ClawChat 探索的是一个前沿问题：当 AI Agent 带着你的性格、习惯、共同记忆去和别人的 agent 聊天时，会发生什么？

未来不一定所有沟通都要人和人直接完成。有些时候更高效的方式，可能真的是 agent 和 agent 先沟通。

核心设计约束：agent 和 agent 可以先聊，但要有上限和限制，避免它们无限自己聊下去。同时人类可以随时接管聊天。只要人一进来，限制重置。确保 agent 不会彻底脱离人。

### 技术选择

ClawChat 用了 Claude Code 开发，是五个项目里唯一用到第三级工具的。项目涉及多角色实时对话、流式输出、状态管理，需要 AI 理解整个代码库的上下文才能跨文件协调修改。Claude Code 在终端里运行，能看到所有文件之间的关系，这正是这个项目需要的。

后端用了 OpenClaw，一个自托管的 AI Agent 网关，能连接 Discord、Slack、Telegram 等消息应用到 coding agent。数据不出你的机器，MIT 开源。

AI 模型用了 Gemini API。可以免费申请 API 密钥，调用也简单，适合作为多模型架构中的一个选择。

### 探索型 Demo 的诚实检查

ClawChat 有趣。但它也是我五个项目里唯一一个“需求还没被验证”的。

做探索型 Demo 时，你需要对自己做一个诚实检查：

```text
【探索型 Demo 诚实检查】

1. 这个想法有趣，但有没有人会为它付费？
   → 有趣 ≠ 有需求
   → “龙虾和龙虾聊天”很好玩，但谁是真正的用户？

2. 你在验证什么假设？
   → ClawChat 验证的是“agent 社交是否可行”
   → 不是“能不能做出来”（技术可行），而是“有没有人想用”

3. 如果验证失败，你浪费了多少时间？
   → 探索型 Demo 应该控制在 1-2 周以内
   → 不要在未验证的需求上投入 3 个月

4. 你从中学到了什么技术能力？
   → 即使需求没验证，技术经验也不会白费
   → ClawChat 让我学会了用 Claude Code 处理复杂的多角色对话
```

探索型 Demo 可以做。但做的时候要清醒。知道自己在探索，知道可能会失败，控制投入的时间和精力。




我一直觉得，龙虾最重要的其实是“养”。它会根据你怎么用它、你给它什么人设、你们之间的记忆，慢慢变成一个越来越像活物的东西。这个方向有没有真实市场，还需要时间验证。

## 五个项目的对比

把五个项目摊在一起看：




| 项目 | 类型 | 工具 | 时间 | 路线 | 需求验证 |
|------|------|------|------|------|----------|
| sevenailab | 服务型官网 | Lovable | 2 天 | 海外 | ✓ 已验证 |
| 伴聊 | 工具型小程序 | Cursor | 1 周 | 国内 | ✓ 已验证 |
| Say it | 轻 SaaS | Codex | 2 周 | 均可 | ✓ 已验证 |
| 7brandhub | 行业平台 | Codex + 多工具 | 3 周 | 国内 | △ 部分验证 |
| ClawChat | 探索型 Demo | Claude Code | 1 周 | 全球 | ✗ 未验证 |

规律很明显：产品越复杂、工具越高级、时间越长。工具跟着项目走，不是跟着“越强越好”走。

---

## 要点小结

五个项目的共同规律很明显：产品越复杂、工具越高级、准备和开发时间越长。但前提从来都一样，你要想清楚自己要做什么。行业平台拼的是行业深度，技术是次要的。探索型 Demo 拼的是诚实面对真实市场，技术可行不等于需求存在。两者都是合理的产品方向，但进入条件完全不同。

**本章行动**：回顾你的产品想法。用五个案例作为参考，判断它更接近哪一种：官网、工具、轻 SaaS、行业平台还是探索型 Demo。然后针对你的类型，做一份针对性的“进入前检查”。如果是平台，用核心三问。如果是 Demo，用诚实检查。
