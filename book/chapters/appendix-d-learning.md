## 一、第二个产品再学：Superpowers 工作流

Superpowers 工作流（github.com/obra/superpowers）适合你开始做第二个、第三个产品时再看。

它的核心不是某个插件，而是一种做事顺序：先写清楚目标和计划，再让 AI 按计划执行，最后用检查清单验收。你不再一条一条随手发指令，而是把“规划、执行、检查”变成稳定流程。

什么时候适合学：

```text
□ 你已经做完第一个产品
□ 你开始频繁重复“先规划、再执行、再检查”
□ 你发现自己需要让 AI 连续处理多个任务
□ 你希望每次开发都有明确的 spec 和验收标准
```

什么时候不需要学：

```text
□ 你还没发布第一个产品
□ 你现在只需要做一个页面或一个小功能
□ 你连需求都没想清楚
```

## 二、Claude Code：最先学的五个动作

第十一章讲过 Claude Code 是命令行型的 AI 编程工具，能理解整个代码库。如果你决定从 Cursor 升级到 Claude Code，不需要读完所有文档，先学这五个动作就够启动了：

**动作 1：`claude`**
在终端里进入你的项目目录，输入 `claude`，就启动了一个对话。你可以直接用自然语言问它代码库的问题，比如“这个项目的入口文件在哪”“用户登录的逻辑是怎么走的”。它会自动扫描你的文件来回答。

**动作 2：引用文件**
在对话里引用具体文件。不同版本的入口和语法可能会变化，有的工具支持 `@文件名`，有的更适合直接说文件路径。比如“帮我看看 src/pages/index.tsx 的首屏部分，有没有可以优化的地方”。这样 AI 就知道你在说哪个文件，不用猜。

**动作 3：运行命令**
让 Claude Code 帮你运行终端命令。比如 `npm run build` 检查代码能不能编译。不同版本对命令触发方式和审批方式会调整，实际使用时以 Claude Code 当前界面提示为准。

**动作 4：`/init`**
第一次在项目里启动 Claude Code 时，可以用初始化命令生成项目说明文件。常见做法是运行 `/init`，让它扫描项目结构，生成一份 CLAUDE.md 文件（下面会讲）。如果命令名称或入口更新，以官方文档为准。这一步帮 AI 建立对项目的初始理解。

**动作 5：`CLAUDE.md`**
这是 Claude Code 的项目说明文件，作用类似第十二章讲的项目 Rules。写在项目根目录，Claude Code 会把它作为项目上下文读取。你可以在里面写项目背景、技术栈、编码规范、不能做的事。

这五个动作足够你启动 Claude Code 的日常使用。用熟之后，你可以再探索更多高级功能，比如 subagents、跨文件重构、自动化测试。具体能力更新很快，正式使用前最好看一眼 Anthropic 官方文档。

## 三、MCP：让 AI 连接外部世界

第十二章讲了 Rules 怎么让 AI 记住项目规范。MCP（Model Context Protocol）做的是另一件事：让 AI 连接你的外部服务。

一句话解释：**MCP 是一个标准协议，让 AI 工具可以读写外部数据源——GitHub 仓库、数据库、API 服务、文件系统，甚至你的浏览器。**

没有 MCP 的时候，Claude Code 只能处理你本地的代码文件。有了 MCP，它可以直接读你 GitHub 上的 issue、查你 Supabase 数据库里的数据、调用你接入的第三方 API。

你不需要现在就学 MCP。但知道它的存在很重要，因为它代表了一个趋势：AI 工具正在从“帮你写代码”变成“帮你操作整个工作流”。当你的产品需要连接多个服务时（比如用户注册后自动发邮件、支付成功后自动更新数据库），MCP 会让这些连接变得更自然。

## 四、Skills：可以带走的技能包

第十二章讲了 Rules 是“这个项目的长期合同”。Skills 是更上一层：**可以跨项目复用的技能包。**

举个例子。你在做 Say it 的时候，写了一套“怎么调用 AI 模型做内容生成”的 Rules。后来你做第二个产品，也需要调用 AI 模型。如果只有 Rules，你得把上一个项目的 Rules 复制过来再改。如果用 Skills，这套“AI 内容生成”的技能可以直接在新项目里激活，不用重写。

Skills 的概念不复杂：Rules 是项目级的，Skills 是个人级的。你积累的每一个 Skill，都在降低你做下一个产品的成本。

## 五、Subagents：从一个 AI 到一组专门角色

第十三章讲了互审三角色：Writer 写、Reviewer 审、你来决策。Subagents 是这个思路的延伸：**不是让一个 AI 什么都做，而是给不同任务配置不同的专门角色。**

Claude Code 支持 subagents。你可以为代码审查、测试、文档、特定技术栈配置专门的子代理。比如做一个有前端、后端、数据库的项目：一个子代理专门审前端组件，一个子代理检查后端接口，一个子代理关注数据库结构。它们能帮助你分工检查，但最终怎么采纳、怎么合并，仍然需要你或主会话来决定。

这听起来很酷，但你大概率还用不到。Subagents 更适合大项目、复杂审查和专项任务。当你的项目大到“一个 AI 同时理解所有文件变得很慢”，或者你需要固定的 reviewer / tester / docs writer 时，再考虑它。

## 六、继续学习的入口

这本书教你“怎么想清楚、怎么做出来”。如果你读完想继续深入工具链，以下是几个推荐的方向：

**Anthropic 官方文档。** Claude Code 和 MCP 的官方文档写得很清楚，有大量示例。如果你要正式使用这些工具，建议从官方文档开始，而不是搜索第三方教程。

**GitHub 上的开源项目。** 第十一章讲过“用开源项目加速”的方法。GitHub 上有大量用 Supabase + Next.js + Stripe 搭建的 SaaS 模板。找一个和你想做的产品类似的模板，用 Cursor 或 Claude Code 在上面修改，比从零开始快很多。

---

最后一句话：工具会一直更新，概念会不断冒出来。但这本书教你的核心方法不会过时——想清楚做什么、说清楚怎么做、判断做得对不对。工具只是手段，判断力才是你的。
