# 第十三章 · 互审：让一个 AI 挑另一个 AI 的漏洞

> 所属部分：第三部分 · 进阶，把工具用深  
> 读完本章，你会得到：互审清单 + 一次完整互审记录  
> 本章方法：互审三角色 + 互审检查清单

---

Say it 的注册流程有一个 Bug，我花了两个晚上 debug 超时问题，试了三种改法都不对。最后把代码贴给 Claude 做 review，它很快指出问题：“你没有处理网络超时的 loading 状态，用户会以为页面卡死了。”

写代码的 AI 也会对自己写的东西有盲区。这和人类一样。你写完一篇文章自己检查，往往看不见自己的错别字，因为大脑会自动补全。

---

## 为什么需要互审

你已经走完了“定生修验发察”的完整流程。你知道怎么让 AI 写代码，也知道怎么验收。

但有一个问题很容易被忽略：你一直在和同一个 AI 对话。

同一个 AI 在同一个上下文里，会形成一种“自洽”。它给你写了代码，你问它“这个代码有没有问题”，它会倾向于说“没问题”。因为它刚刚才写的，它的上下文里充满了“这是对的”的暗示。

解决办法很简单：让另一个“人”来看。

在 AI 时代，这个“另一个人”可以是另一个 AI。

---

## 互审三角色

我做 Say it 和 ClawChat 的时候，逐渐形成了一个工作流。我把它叫“互审三角色”：

```text
【互审三角色】

角色 1：Writer（写手）
  → 负责写代码的 AI
  → 在 Lovable/Cursor/Claude Code 里工作
  → 它只管"做出来"

角色 2：Reviewer（审稿人）
  → 另一个 AI，专门挑问题
  → 你把代码贴给它，让它找漏洞
  → 它只管"挑毛病"

角色 3：PM（产品经理）
  → 你
  → 你看 Writer 的产出，看 Reviewer 的意见
  → 你决定采纳哪些、忽略哪些、优先修什么
```

Writer 负责执行。Reviewer 负责质疑。你负责判断。




这个分工的关键是，写和审最好不要由同一个 AI 在同一个对话里完成。你需要切换上下文。

最简单的做法：在 Cursor 里写代码，写完后把关键代码复制到 Claude 对话版（或 ChatGPT）里，让它 review。它没有“我刚写的”这个心理包袱，能更客观地找到问题。

但我后来发现，有一种更省时间的互审方式：**写代码之前就互审**。

具体做法是：先开启计划模式（Cursor 的 Plan mode 或者直接让 AI 写一份开发计划），让负责写代码的 AI 先输出一份完整的开发计划，包括要改哪些文件、每个文件改什么、改动的先后顺序。然后把这份计划复制到另一个 AI 窗口，让它 review 这个计划：“这个开发计划有什么遗漏？逻辑上有没有漏洞？有没有更好的实现方式？”

review 完、把问题修正后，再让第一个 AI 按修正后的计划执行。

这个方法的好处是：在写代码之前就把弯路堵死了。比写完代码再找 Bug 高效得多，因为改计划只需要改几句话，改代码可能要改几十个文件。

更高级的做法：用 Claude Code 写代码，用 Claude 对话版 review 思路。或者反过来，用 Claude 对话版规划方案，用 Claude Code 执行，执行完再让对话版检查结果。

## 互审检查清单

让 Reviewer 审什么？不要只说“看看有没有问题”。你需要给它一个具体的检查维度。

```text
【互审检查清单】

功能完整性：
  □ 核心流程能从头走到尾吗？
  □ 有没有承诺了但没实现的功能？
  □ 边界情况有没有处理？（空输入、超长文本、特殊字符）

错误处理：
  □ 表单提交失败时用户看到什么？
  □ API 调用超时有没有兜底？
  □ 按钮重复点击有没有防护？

安全性：
  □ API key 有没有暴露在前端代码里？
  □ 用户输入有没有做验证和清洗？
  □ 环境变量是不是正确配置？

性能：
  □ 图片有没有压缩？
  □ 不必要的重复渲染有没有？
  □ 首屏加载速度可接受吗？

用户体验：
  □ 加载状态有没有提示？
  □ 操作成功后有没有反馈？
  □ 移动端能不能正常使用？
```

你不需要每次都把整个清单跑一遍。但每次互审至少覆盖 2-3 个维度。

也不要以为只有代码才能被审。对新手来说，更推荐先审三类东西：




```text
【三种互审对象】

1. 审计划
   → 在写代码前，把 PRD、开发计划、页面结构发给 Reviewer
   → 让它检查：需求有没有漏、流程有没有绕、有没有过度设计

2. 审结果
   → 把截图、URL、用户流程描述发给 Reviewer
   → 让它检查：用户看不看得懂、移动端有没有问题、CTA 是否明确

3. 审代码
   → 只有在你需要处理具体 Bug、安全、性能时，再贴关键代码或关键文件
   → 让它按检查清单逐条看
```

把这个清单连同你的计划、截图或代码一起发给 Reviewer AI：“请按照以下清单检查，列出发现的问题，并按严重程度排序。”它会逐条给你反馈。

最值钱的是“审计划”。因为计划阶段改一句话就够了，等代码写完再发现方向错，可能要重做几十个文件。

## 上线前安全五问

互审里有一类问题不能靠感觉，那就是安全。

你不需要变成安全工程师，但上线前至少问五个问题：




```text
【上线前安全五问】

1. API key 有没有写进前端代码？
   → Secret key 只能放在 `.env.local`、服务端或部署平台环境变量里，不能出现在公开代码中。部分 public/anon key 可以放前端，但仍要配合权限策略。

2. .env 文件有没有被上传到 GitHub？
   → 检查 `.gitignore` 里有没有 `.env`、`.env.local`。

3. 用户输入有没有限制？
   → 空输入、超长文本、特殊字符，都要有基本处理。

  4. 支付回调有没有验证？
   → 支付成功不能只看前端跳转，要以后端或服务商回调为准，并验证签名。

5. 错误信息有没有泄露敏感信息？
   → 不要把密钥、数据库地址、服务器路径直接显示给用户。
```

这五问不解决所有安全问题，但能挡住新手最常见的坑。尤其是 secret key 泄露和支付回调，这两件事一旦出问题，不只是页面难看，而是真金白银的损失。

## Lovable 的内置互审

如果你还在用 Lovable，它其实已经有一种内置的“互审”机制。

Lovable 发布前会自动扫描你的网站，给出安全和速度报告。比如“图片太大，影响加载速度”或“这个 API 调用有安全隐患”。

最关键的是：你不需要懂这些报告在说什么。点“Try to fix”，它自己修。

这就是最轻量的互审。同一个工具里，一个 Agent 写代码，另一个 Agent 检查问题。你只需要决定“修不修”。

但这种内置互审有局限。它只检查技术问题（速度、安全），不检查业务逻辑（功能对不对、流程通不通）。业务层面的互审，还是需要你手动切换到另一个 AI。

---

## Claude Code 的审批机制

如果你用 Claude Code，它通常会有审批或权限机制：执行命令、修改文件前，会根据你的设置请求确认。

你可以逐个审批改动，看它要改什么，同意或拒绝。也可以开启“Accept all”模式让它自己跑。

这个机制本身就是一种互审。AI 提议，你审批。即使自动化程度再高，关键决策权始终在你手里。

但注意：自动接受改动的模式虽然快，但你放弃了逐条审查的机会。在关键功能的开发阶段（比如支付流程、用户数据处理），建议关闭自动接受，一条条看。

## 一次真实的互审记录

给你看我做 Say it 时的一次互审过程，作为参考。

我在 Cursor 里写完了“语音转文字 → 生成三种内容”的核心流程。看起来能跑。

然后我把核心代码贴到 Claude 对话版里，说：

“请检查这段代码的错误处理和用户体验。特别关注：API 调用失败时的兜底、loading 状态、移动端表现。”

Claude 给了我五条反馈：

1. 语音上传没有文件大小限制，用户传一个 100MB 的文件会让 API 超时。
2. 转文字过程中没有 loading 提示，用户会以为页面卡了。
3. 生成的三种内容没有“复制”按钮，用户需要手动选中复制。
4. 如果 AI 生成的内容为空（API 返回了空字符串），页面会显示一个空白卡片。
5. 移动端的“开始录音”按钮太小，难以点击。

五条里，第 1、2、4 条是我完全没想到的。第 3 条是“做了但忘了加”。第 5 条验证了我的一个隐约担心。

我把这五条整理成修改清单，回到 Cursor 里一次性修复。

整个互审花了 10 分钟。但它帮我避免了上线后才被用户发现这些问题。

## 互审的节奏

互审不需要每一次修改都做。那太频繁了。

我的节奏是这样的：

写完一个完整功能后，互审一次。一个功能模块完成后，比如“注册流程做完了”“支付流程接好了”“首页重新设计了”，做一次就够。

发布前，做一次全面互审。把上线前检查清单和互审检查清单合在一起走一遍。

收到用户反馈后，做一次针对性互审。用户说“这个地方不好用”，你让 Reviewer AI 专门检查那个部分。

不需要更多了。三个时机，足够覆盖大部分问题。

如果你后面用到 Claude Code，它还有 subagents 这类更进阶的玩法：给代码审查、测试、文档等任务配置专门角色，让它们分工检查。这是“互审”的延伸版——不只是一个写一个审，而是多个专门角色各司其职。但最终怎么采纳、怎么合并，仍然需要你或主会话来决定。附录 D 会简单介绍这个概念。

## 要点小结

一个 AI 写的代码，让同一个 AI 在同一个上下文里自查，容易沿着原来的思路继续自洽。切换到另一个上下文，用“互审三角色”（Writer 写、Reviewer 审、PM 决策）能发现更多盲区。关键是不同的人（或 AI）看问题的角度不一样。

**本章行动**：把你产品的核心功能代码复制到新的 AI 对话，用互审检查清单让它找 3-5 个问题。记录反馈，标记哪些需要修，回到开发工具一次性改完。
