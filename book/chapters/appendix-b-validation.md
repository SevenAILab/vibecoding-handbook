## 二、需求强度判断表

```text
这个需求属于哪一类？

□ 痛点
  用户现在就难受，不解决会继续难受。
  证据：____________________________________

□ 爽点
  用户用了之后明显更快、更省事、更舒服。
  证据：____________________________________

□ 痒点
  有也挺好，没有也能活。
  证据：____________________________________

判断规则：
有替代方案 = 至少有真实需求。
有人花钱或花大量时间解决 = 需求更强。
没人行动，只是说“挺有意思” = 大概率是痒点。
```

## 五、快速判断清单

```text
1. 这个需求是痛点、爽点还是痒点？
   → 痛点：优先做。爽点：可以做，但要让用户快速体验到。痒点：第一版别做太重。

2. 用户现在用什么替代方案？
   → 有替代方案 = 需求已验证，你要做的是更好。没有 = 需求可能不够强。

3. 用户雇这个产品完成什么任务？
   → 用一句话说清楚。说不清楚就还没想明白。

4. 去掉 AI，这个产品还成立吗？
   → 成立 = AI 是加速器，需求本身是真的。不成立 = 可能在为了 AI 而 AI。

5. 有没有至少 1 个人告诉你他遇到过这个问题？
   → 有 = 可以开始做最小版本。没有 = 先去找人聊。
```
