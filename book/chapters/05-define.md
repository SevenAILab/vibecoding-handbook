# 第五章 · 定：把想法翻译成 AI 能执行的任务

> 所属部分：第二部分 · 主控流程，定生修验发察  
> 读完本章，你会得到：一份 MVP 需求文档  
> 本章方法：PRD 三段式 + Plan mode

---

开战卡写完了。做什么、给谁做、第一版做什么、不做什么，都有了。

问题来了：AI 看不懂开战卡。

这一章要做的，是把开战卡翻译成一份“AI 能执行的任务书”。在产品世界里，这东西叫 PRD（Product Requirements Document），也就是产品需求文档。

别被这个词吓到。

它不需要像大公司里那种 20 页的文档。

它更像你去蛋糕店订蛋糕时填的那张单子。

---

## 为什么要先写需求文档

常见做法是：想到一个点子，直接打开 Lovable 或 Cursor，开始让 AI 写。

这能跑。上一章你就是这么做的。

但如果你要做的是一个有交互、有逻辑、有用户流程的产品，“直接让 AI 写”会出问题。

我做 Say it 的时候踩过这个坑。最开始想很多功能：很多页面、用户设置、让用户自己设定提示词、先把所有东西都配齐。越做越发现，第一版真正需要的其实只有一个最小闭环：开始对话 → 完成对话 → 生成三种内容。

除此之外的所有功能都砍了：登录后才能试用？砍。用户档案设置？砍。自定义提示词？砍。选择语音交流时间？砍。

不要多余页面，不要太重的设置，不要先做那些“以后可能有用”的东西。

如果一开始没想清楚，AI 会特别热情地帮你把错误方向越做越完整。我的教训是，上来先写清楚三件事：这个产品一句话是什么、它做什么、它不做什么。“不做什么”和“做什么”一样重要，因为 AI 太会做加法了。

所以在让 AI 动手之前，先花 30 分钟写一份需求文档。

准备工作做得越充分，后面开发越顺利。急着写代码，结果写到一半发现逻辑不对，又要推倒重来。

---

## 写 PRD 之前，先把模糊变具体

不过在写需求文档之前，先做一件事：把你脑子里那句模糊的话拆干净。

“我想做一个帮人把说话变成文章的工具。”——我一开始对 Say it 的描述就是这样的。听起来好像已经够具体了。但这句话喂给 AI，它只能给你一个模糊的东西。因为这句话里的每个词，都还没想清楚。

“说话”——是实时录音还是导入音频文件？是只支持中文还是多语言？“变成文章”——是逐字转写还是自动整理成段落？要不要加标题和分段？“工具”——是网页还是 App？要不要登录？数据存在哪里？

你把这些问题一个个回答完，那句模糊的话就会变成：“一个网页工具，用户点击录音按钮用中文说话，结束后自动整理成带标题和段落的文章，支持一键复制，不需要登录。”




这才是 AI 能听懂、能执行的描述。

方法很简单：拿你开战卡上那句产品定义，每遇到一个关键词就问自己“到底什么意思”。“文档处理”是 PDF 还是 Word？“提高效率”是速度快还是质量高？“用户”是学生还是上班族？每拆一层，描述就具体一层。拆到你自己觉得没有歧义了，PRD 也就自然出来了。

然后你把它往下面的三段式里一填就行。

---

## PRD 三段式

我不推荐你写大公司那种 PRD。

我推荐一个“三段式”，够用就行：

```text
【第一段：这是什么】
一句话产品定义：____
给谁用：____
为什么用：____
用户的关键动作是什么：____

【第二段：第一版做什么】
核心功能（不超过 3 个）：
  1. ____
  2. ____
  3. ____
用户流程（从打开到完成）：
  打开页面 → ____ → ____ → ____ → 完成
这一版不做什么：____

【第三段：技术约束】
技术栈偏好：____（不确定就写"AI 推荐"）
是否需要数据库：____
是否需要登录：____
是否需要支付：____
部署方式：____（Lovable 自动部署 / Vercel / 不确定）
```

第一段解决“为什么做”。




第二段解决“做什么、不做什么”。

第三段解决“技术边界”。你可能不懂技术，但你要告诉 AI：这一版暂时不需要数据库，先存本地（localStorage）；不需要登录；不需要支付。这样 AI 就不会过度设计。

技术栈那一行如果你完全不懂，可以先写“使用 Next.js + TypeScript，暂时不需要数据库”。这是目前 AI 开发工具最顺的默认配置之一，后面我们会讲为什么。不确定填什么技术？先写“不确定，AI 推荐”就行。技术栈是 AI 的事，你的事是讲清楚产品要解决什么问题。

---

## 让 AI 帮你写需求文档

你不用自己从零写这份文档。

最好的方式是：你先把开战卡里的信息告诉 AI，让它帮你生成一份 MVP 需求文档，然后你自己审。

我做伴聊的时候用过这个方法。给 Claude 的 prompt 大概是：

“基于我之前的调研报告，帮我写一份完整的小程序需求文档，包括：产品定位和目标用户、核心功能列表和优先级、每个功能的详细交互流程、页面结构和跳转逻辑、数据库表结构设计、技术实现方案。用 Markdown 格式输出，结构清晰。”

这份文档会成为整个开发的基础，避免开发时走偏。

你不用写这么复杂。简化版是这样：

```text
我要做一个____（从开战卡里复制你的产品定义）。

目标用户：____（从三秒场景卡里复制）。
核心场景：____。

请帮我写一份 MVP 版本的需求文档，包括：
1. 一句话产品定义
2. 核心功能（不超过 3 个）和优先级
3. 用户流程（从打开到完成）
4. 页面结构
5. 这一版明确不做什么

技术约束：使用 Next.js + TypeScript，暂时不需要数据库，
先用 localStorage 存储。
```

AI 生成的需求文档你要自己看一遍。

重点检查三件事：

1. 它有没有加了你不需要的功能？（AI 特别爱加注册登录、用户设置、深色模式这些你没要的东西。）
2. 用户流程是不是你想的那样？从打开到完成，最短要几步？
3. “不做什么”有没有写清楚？

如果 AI 加了太多东西，大方地砍。

用户第一次打开网页最短要点几下，才能拿到价值。把这个路径压到最短。

## 用 Plan mode 让 AI 先问你

Lovable 有一个特别好的功能叫 Plan mode。

普通模式（Agent mode）是你说什么，它直接做。

Plan mode 是你说了之后，它先不做，而是反过来问你问题，比如“你是做给谁用的？”“你希望用户登录吗？”“你想用什么颜色风格？”

这些问题特别有用。

因为很多时候你以为自己想清楚了，但 AI 的追问会让你发现自己漏掉了什么。

Lovable 官方也建议：先用 Plan mode 想清楚，再切换到 Agent mode 执行。

具体操作：打开 Lovable 的编辑器，在输入框旁边切换到 Plan mode。把你的需求文档贴进去，让它提问。

你可能会收到这样的问题：

“用户是否需要注册？”不需要，先让用户直接用。
“数据需要持久化吗？”第一版不需要，先存浏览器本地。
“需要移动端适配吗？”需要。
“你想用什么配色方案？”简洁，参考____。

回答完这些问题后，再切换到 Agent mode，让 AI 正式开始生成。




MVP 需求文档应该经过多轮沟通才能定型。不要指望一轮就完美。跟 AI 对话 3-5 轮，让需求越来越清楚。

## 一个完整的启动 prompt 长什么样

做完需求文档和 Plan mode 问答后，你就可以把最终的需求喂给 AI 了。

我给你看一个完整样本。这是我做 sevenailab 时的启动 prompt 的结构：

```text
帮我创建一个为中国出海品牌提供 GEO 服务的介绍网站。

业务定义：帮品牌在 AI 答案里更容易被看见。
目标用户：中国出海品牌的市场负责人。
页面清单：主页、服务介绍、案例、工作流程、关于我们、联系我们。
关键动作：首页有一个"预约 GEO 诊断报告"的 CTA 按钮。
语言策略：中英双语，英文为主，中文为辅。
审美锚点：参考苹果的极简设计风格。
不要什么：不要紫色赛博风，不要像 AI 公司模板。
技术约束：暂时不接数据库，表单提交先用 Lovable 自带方案。
```

你看，这里面有：业务定义（给谁、做什么）、页面清单（6 个模块）、关键动作（CTA 按钮）、语言策略、审美锚点（苹果风格）、反面参照（不要什么）、技术约束。

一条 prompt 把“问题定义”压得很密。

你的第一条 prompt 不用这么密。但方向是一样的：越具体，AI 的第一版越对，后面修改的次数越少。

像 CEO 一样沟通，不要说“帮我写个网站”。要说完整的背景、任务、数据、参考、限制。

---

## 如果卡住了

写 PRD 最容易卡在“核心功能”这一步。你觉得有三个功能都很重要，砍不动。

试试这个问题：如果你只能留一个功能，哪个功能没了产品就不成立？那个就是核心功能。其他的放到“不做什么”里。

还有一种常见的卡法：你不知道技术栈该写什么。不知道就空着，或者写“交给 AI 推荐”。技术栈那一栏不影响 Plan mode 的问答。AI 会根据你的功能描述推荐合适的方案。

---

## 要点小结

想清楚需求，是键盘前最值钱的工作。一份好的 PRD 不是形式主义，它是防止 AI 把错误方向越做越完整的第一道防线。用 PRD 三段式把“这是什么”“做什么”“怎么做”压进一份文档，让 AI 在 Plan mode 里反复提问澄清，然后才切到正式开发。

**本章行动**：打开你的开战卡，用 PRD 三段式写一份需求文档（第一段：产品定义+给谁+为什么，第二段：核心功能+用户流程+不做什么，第三段：技术栈+数据库+部署），把它贴进 Lovable 的 Plan mode 让 AI 提问，回答 3-5 轮直到需求清楚。如果你想看更完整的需求验证方法，附录 B 有一份工具箱可以参考。
